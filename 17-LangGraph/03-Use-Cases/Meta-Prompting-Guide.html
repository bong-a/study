meta-prompting · 학습 노트
OpenAI · Prompt Generation

프롬프트를 직접 쓰지 말고,
프롬프트를 써주는 프롬프트에게 시켜라

이것이 메타 프롬프트 엔지니어링입니다. 당신은 "하고 싶은 일"만 설명하면, 프롬프트 엔지니어링 모범 사례가 모두 반영된 완성된 프롬프트가 자동으로 만들어집니다.

META-PROMPT · 셰프
"좋은 프롬프트를 만들어라"고 모델에게 지시하는 상위 프롬프트
PROMPT · 레시피
모델에게 주는 실제 지시문 (자동 생성됨)
OUTPUT · 요리
모델의 최종 응답
"meta-" = ~에 대한 ~  ·  프롬프트를 만들기 위한 프롬프트

01 핵심 개념

먼저 세 단어만 구분하면 끝

모든 혼란은 이 세 가지를 헷갈리는 데서 시작합니다. "메타(meta-)"는 "~에 대한 ~"라는 뜻 — 즉 프롬프트를 만들기 위한 프롬프트, 스키마를 만들기 위한 스키마입니다.

용어정체비유
프롬프트
Prompt		모델에게 주는 실제 지시문🍳 요리 레시피
메타 프롬프트
Meta-Prompt		"좋은 프롬프트를 만들어라"고 모델에게 지시하는 상위 프롬프트👨‍🍳 레시피를 써주는 셰프
메타 스키마
Meta-Schema		"좋은 JSON 스키마를 만들어라"고 지시하는, 자기 자신을 설명하는 JSON📋 양식을 만들어주는 양식

02 왜 필요한가

노하우를 몰라도, 전문가 수준 프롬프트를 얻는다

기존 방식은 사람이 프롬프트 엔지니어링 모범 사례를 직접 알아야 했습니다. 메타 방식은 그 노하우를 모델 안에 내장시켜버립니다.

기존 방식

사람이 직접 작성

사람 → 프롬프트 직접 작성 → 모델
모범 사례를 스스로 알고 있어야 하고, 손이 많이 감
메타 방식

할 일만 설명

사람 → 할 일 설명만 → [메타 프롬프트 + 모델] → 완성된 프롬프트
모범 사례가 자동 반영됨

03 동작 원리

5단계 워크플로우

입력부터 전달까지, 시스템 내부는 이렇게 흘러갑니다.

04 생성 규칙

생성된 프롬프트가 반드시 지키는 원칙

③ 생성 단계에서 메타 프롬프트는 결과물이 다음 규칙을 따르도록 강제합니다.

원칙내용왜?
추론 먼저,
결론 나중		이유를 먼저 설명하게 하고, 결론·분류·결과는 항상 맨 마지막에답부터 성급히 내면 정확도↓ (Chain-of-Thought)
예시 포함1~3개의 고품질 예시 제공원하는 형식을 모델이 흉내 내게 함
플레이스홀더복잡한 부분은 [대괄호]로 표시예시를 재사용 가능한 템플릿으로
출력 형식 명시원하는 결과 포맷을 상세히 지정일관된 결과 보장
구조화된 섹션예시·단계·주의사항·출력형식을 구획으로 분리지시가 명확해짐
KEY

가장 중요한 규칙 — "결론은 항상 마지막에". 모델이 답을 내기 전에 생각할 시간(추론 토큰)을 주는 것이 메타 프롬프트의 핵심 철학입니다.

05 구조화 출력

스키마 생성과 재미있는 딜레마

JSON 형태의 정형 출력을 만들 때는 메타 스키마를 씁니다. 생성된 스키마는 다음 제약을 지켜야 합니다.

규칙값
모든 필드required (필수)로 지정
additionalPropertiesfalse — 정의 안 된 필드 금지
객체(object)반드시 properties를 가져야 함
필드 순서추론 → 결론 순서 유지

⚠️ "닭이 먼저냐 달걀이 먼저냐" 딜레마

메타 스키마는 동적으로 키를 생성해야 해서 additionalProperties에 의존하는데, 이건 OpenAI의 strict 모드에서는 금지된 기능입니다.

[모순] strict 모드로 스키마를 만들고 싶다 ↓ 그런데 스키마를 "만드는 과정"은 strict 모드가 지원 안 하는 기능이 필요하다 ↓ [해결] "유사 메타 스키마(pseudo-meta-schema)" 사용 ↓ 생성은 non-strict로 → 생성 후 ④검증 단계에서 strict 모드에 맞게 자동 정리(cleaning)
한 줄 요약: "만들 때는 느슨하게, 다 만든 뒤 엄격하게 정리"하는 방식으로 우회합니다.

06 코드 패턴

실제 호출은 이 두 가지

두 예제의 패턴은 동일합니다 — system에 메타 프롬프트/메타 스키마를 넣고, user에 "내가 하고 싶은 일"만 설명합니다.

① 텍스트 프롬프트 생성.py
from openai import OpenAI

client = OpenAI()

def generate_prompt(task_or_prompt: str):
    completion = client.chat.completions.create(
        model="gpt-5.5",
        messages=[
            {"role": "system", "content": META_PROMPT},          # ← 핵심: 메타 프롬프트를 system 역할로
            {"role": "user",   "content": "Task, Goal, or Current Prompt:\n" + task_or_prompt},
        ],
    )
    return completion.choices[0].message.content
② 스키마 생성.py
def generate_schema(description: str):
    completion = client.chat.completions.create(
        model="gpt-5.4-mini",
        response_format={"type": "json_schema", "json_schema": META_SCHEMA},  # ← 메타 스키마 지정
        messages=[
            {"role": "system", "content": META_PROMPT},
            {"role": "user",   "content": "Description:\n" + description},
        ],
    )

07 두 가지 특수 모드

기존 프롬프트 "개선" & 오디오 출력

✏️ 기존 프롬프트 개선 모드

새로 만드는 것뿐 아니라 기존 프롬프트를 다듬을 수도 있습니다. 이때는 모델이 "어디를 왜 고칠지" 먼저 스스로 진단한 뒤 개선안을 내놓습니다.

<reasoning> - 변경이 간단한가? - 추론(Chain-of-Thought) 순서가 올바른가? - 구조가 좋은가? - 예시가 대표성이 있는가? - 복잡도는 적절한가? </reasoning> ↓ 최종 개선된 프롬프트 출력

🔊 오디오 출력용 특수 규칙

음성으로 답하는 모델용 프롬프트에는 별도 제약이 추가됩니다.

항목규칙
길이짧고 대화체로 (문장당 5~20 단어 정도)
톤감정 있고 친근하게 (emotive & friendly)
예시여러 턴의 자연스러운 대화 흐름 포함
속도사용자를 기다리게 하지 않도록 빠른 응답

08 정리

이 문서가 말하는 것

"프롬프트 엔지니어링 노하우"를 모델 안에 내장시켜서 —
사용자는 노하우를 몰라도, 할 일만 말하면 전문가 수준 프롬프트를 얻는다.

  • 1한 단계 위에서 생각하기. 프롬프트를 직접 쓰지 말고, 프롬프트를 써주는 프롬프트를 쓴다.
  • 2모범 사례의 자동화. "추론 먼저·결론 나중", "예시 포함", "형식 명시" 같은 규칙이 자동 적용된다.
  • 3생성 후 검증. 특히 스키마는 느슨하게 생성 → strict 모드로 정리하는 2단계 안전장치를 둔다.
