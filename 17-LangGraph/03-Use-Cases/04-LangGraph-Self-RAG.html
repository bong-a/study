검색한 문서도, 스스로 만든 답도 다시 검증하는 RAG

Self-RAG — 검색한 문서가 질문에 맞는지 채점하고, 답을 만든 뒤에도 그 답이 근거에 기반하는지(환각) · 질문을 실제로 해결하는지 다시 채점합니다. 미흡하면 질문을 고쳐 재검색하거나 다시 생성하는 자기 반성(self-reflection) RAG입니다.

일반 RAG는 검색 → 생성 한 방향으로 끝납니다. Self-RAG는 이 흐름 곳곳에 자기 점검(채점) 관문을 세워 두고, 통과하지 못하면 앞 단계로 되돌아갑니다.

점검은 두 지점에서 일어납니다. ① 생성 전 — 검색한 문서가 질문과 관련 있는지 채점하고, 쓸 문서가 없으면 질문 재작성 → 재검색. ② 생성 후 — 만든 답이 근거에 기반하는지(환각 여부)와 질문을 해결하는지 채점하고, 실패하면 재생성 또는 재검색. 이 되돌아가는 루프가 Self-RAG의 핵심입니다.

01Self-RAG의 네 가지 자기 판단

원 논문(Self-RAG, 2310.11511)은 RAG 과정에 네 번의 스스로 판단하는 단계를 제안합니다. 이 튜토리얼은 그중 세 가지를 yes/no 이진 채점으로 단순화해 구현합니다.

자기 판단 입력 → 출력 (원 논문) 이 튜토리얼 구현
검색 필요? 질문 → yes / no / continue 생략 (항상 먼저 검색)
문서 관련성 질문+문서 → relevant / irrelevant yes/no · GradeDocuments
사실성 (환각) 질문+문서+답변 → 완전/부분/무근거 yes/no · Groundedness
정답 품질 질문+답변 → 5 · 4 · 3 · 2 · 1 yes/no · GradeAnswer

02바로 앞의 CRAG와 무엇이 다른가

둘 다 "검색 결과를 의심하고 채점"하지만, 보정 방법과 점검 범위가 다릅니다. CRAG는 밖(웹)에서 보강하고 한 방향으로 끝나지만, Self-RAG는 안(문서)에서 다시 찾으며 답까지 검증하고 되돌아갑니다.

비교 항목CRAG (03)Self-RAG (04)
점검 대상 검색 문서만 검색 문서 + 생성 답변
문서가 부적합할 때 웹 검색으로 지식 보강 질문 재작성 → 재검색(루프백)
생성 후 점검 없음 (단방향으로 종료) 환각·정답성 채점 후 재생성/재검색
외부 웹 검색 사용 (Tavily) 사용 안 함 (내부 문서 재검색)
반복 구조 없음 루프 있음 → 무한 루프 주의

04세 개의 채점기 + 질문 재작성기

Self-RAG의 "자기 반성"은 아래 네 개의 LangChain 체인으로 이뤄집니다. 채점기 셋은 모두 with_structured_output으로 yes/no 이진 점수만 강제로 뱉게 만듭니다.

Retrieval Grader

검색된 문서를 한 개씩 질문과의 관련성으로 채점. 생성 전에 쓸모없는 문서를 걸러 냄.

GradeDocuments · binary_score
Hallucination Grader

생성된 답이 검색 문서에 근거하는지(환각 여부) 채점. no면 지어낸 답 → 재생성.

Groundedness · 근거성 점검
Answer Grader

근거는 있어도 질문을 실제로 해결하는지 채점. no면 동문서답 → 재검색.

GradeAnswer · 정답성 점검
Question Re-writer

질문의 숨은 의도를 파악해 검색에 더 잘 맞는 형태로 다시 씀. 두 루프 모두의 되돌아가는 지점.

re_write_prompt | llm | StrOutputParser()

05상태(State) — CRAG보다 단순

Self-RAG는 경로를 가르는 별도 스위치 필드가 필요 없습니다. 분기는 그때그때 채점기를 직접 호출해 결정하므로, 상태에는 딱 세 값만 담깁니다.

question str
사용자 질문. 재작성되면 이 값이 교체되어 다시 검색된다.
generation str
LLM이 만든 답변. 환각 판정 시 덮어쓰며 재생성된다.
documents List
검색·선별된 문서 목록.

06노드 구성

CRAG와 달리 웹 검색 노드가 없고, transform_query가 retrieve로 되돌아갑니다.

노드하는 일갱신하는 키
retrieve PDF Retriever로 질문에 맞는 문서를 검색 documents
grade_documents 문서별로 관련성 채점, 관련된 것만 남김 documents
generate 선별된 문서로 RAG 답변 생성 generation
transform_query 질문을 재작성 → 다시 retrieve로 보냄 question

07두 개의 분기 함수

조건부 엣지가 두 군데 붙습니다. 하나는 생성 전(문서 채점 결과), 하나는 생성 후(답변 채점 결과)입니다.

① decide_to_generate — 생성 전 (문서 채점 후)

남은 문서가 있으면 생성으로, 하나도 없으면 재검색 루프로 보냅니다.

decide_to_generate() · 문서 유무로 분기
def decide_to_generate(state):
    if not state["documents"]:      # 관련 문서 0개
        return "transform_query"     # 질문 재작성 → 재검색
    else:
        return "generate"            # 관련 문서 있음 → 생성

② grade_generation_v_documents_and_question — 생성 후 (2단 채점)

먼저 환각을 점검하고(근거성), 통과하면 정답성을 점검합니다. 세 가지 결말이 나옵니다.

① 환각 점검
(groundedness)		 ② 정답성 점검
(answer)		 반환값 다음 이동
no · 근거 없음 — hallucination generate 재시도
yes · 근거 있음 yes · 질문 해결 relevant END ✓
yes · 근거 있음 no · 못 풂 not relevant transform_query
grade_generation_v_documents_and_question() · 2단 채점
def grade_generation_v_documents_and_question(state):
    # 1차: 환각(근거성) 점검
    grade = groundedness_grader.invoke(...).binary_score
    if grade == "no":
        return "hallucination"    # 근거 없음 → 재생성

    # 2차: 정답성 점검
    grade = answer_grader.invoke(...).binary_score
    if grade == "yes":
        return "relevant"         # 질문 해결 → END
    else:
        return "not relevant"     # 못 풂 → 재검색

08실행 예시 — 정상 종료 vs 무한 루프

루프가 있다는 건 빠져나오지 못할 수도 있다는 뜻입니다. 그래서 recursion_limit으로 최대 반복 횟수를 정해 둡니다.

정상 종료 — 문서에 답이 있는 경우 relevant → END

질문: "삼성전자가 개발한 생성형 AI 의 이름은?"

retrieve → grade_documents → generate → grade_generation → END

PDF에 관련 내용이 있어 문서 채점을 통과합니다. 생성한 답도 근거가 있고(환각 없음) 질문을 해결하므로 relevant 판정 → 곧바로 "삼성 가우스"를 답하고 종료합니다.

무한 루프 — 문서에도 없고 답도 못 만드는 경우 GraphRecursionError

질문: "테디노트가 개발한 생성형 AI 의 이름은?"

retrieve → grade_documents → transform_query ↺ 반복 → recursion_limit 초과

PDF에 없는 내용이라 관련 문서가 계속 0개입니다. 재작성 → 재검색을 아무리 반복해도 문서를 못 찾아 같은 루프를 맴돌다 recursion_limit(=10)에 걸려 멈춥니다. Self-RAG를 실전에 쓰려면 이런 루프를 빠져나올 탈출 로직(예: 웹 검색 대체)이 필요합니다.

09그래프 조립 코드

조건부 엣지가 두 개인 것이 핵심입니다. transform_query는 retrieve로, hallucination은 generate로 되돌아가 루프를 만듭니다.

workflow 구성 · StateGraph
workflow.add_edge(START, "retrieve")
workflow.add_edge("retrieve", "grade_documents")

# ① 생성 전 분기: 문서 유무
workflow.add_conditional_edges(
    "grade_documents", decide_to_generate,
    {"transform_query": "transform_query", "generate": "generate"},
)
workflow.add_edge("transform_query", "retrieve")   # 재검색 루프

# ② 생성 후 분기: 환각·정답성
workflow.add_conditional_edges(
    "generate", grade_generation_v_documents_and_question,
    {
        "hallucination": "generate",       # 재생성 루프
        "relevant": END,                  # 종료
        "not relevant": "transform_query",  # 재검색 루프
    },
)

app = workflow.compile(checkpointer=MemorySaver())

10핵심 정리

  • Self-RAG는 검색 문서와 생성 답변을 모두 채점하는 자기 반성형 RAG다.
  • 점검은 두 지점 — 생성 전(문서 관련성), 생성 후(환각 + 정답성)에서 일어난다.
  • 생성 후 채점은 2단계다: 먼저 근거성(환각), 통과하면 정답성(질문 해결).
  • 실패하면 transform_query → retrieve(재검색) 또는 generate(재생성)로 되돌아간다.
  • CRAG는 웹으로 보강·단방향, Self-RAG는 내부 재검색·루프라는 점이 가장 큰 차이다.
  • 루프가 있으므로 recursion_limit이 필수 — 못 빠져나오면 GraphRecursionError가 난다.
