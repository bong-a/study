LangGraph · 03-Use-Cases · 05

먼저 계획을 세우고, 한 단계씩 실행하며 계획을 고쳐 나가는 에이전트

Plan-and-Execute — 복잡한 작업을 장기 계획으로 먼저 나눈 뒤, 한 단계씩 실행하고 매번 남은 계획을 다시 점검·수정합니다. 매 순간 즉흥적으로 움직이는 ReAct와 달리, 명시적 계획으로 긴 작업에서도 길을 잃지 않습니다.

일반 ReAct 에이전트는 "생각 → 행동"을 매 순간 즉흥적으로 반복합니다. 편하지만 작업이 길어지면 큰 그림을 놓치고 헤매기 쉽습니다.

Plan-and-Execute는 이를 계획과 실행으로 분리합니다.

  • 1계획(planner) — 목표를 보고 전체 단계 리스트를 먼저 만든다.
  • 2실행(execute) — 계획의 첫 단계 하나를 ReAct 에이전트로 수행한다.
  • 3재계획(replan) — 결과를 보고 남은 계획을 다시 세우거나, 다 됐으면 종료를 결정한다. → 2로 반복.
  • 4최종 보고서(final report) — 쌓인 결과를 정리해 마크다운 보고서로 마무리한다.

01Plan-and-Execute란 & ReAct와 무엇이 다른가

Plan-and-Solve 논문(2305.04091)과 Baby-AGI에서 영감을 받은 방식으로, 명시적인 장기 계획을 강조한다는 점이 핵심입니다. 전통적 ReAct와 비교하면 차이가 뚜렷합니다.

항목 ReAct 스타일 Plan-and-Execute
진행 방식 한 번에 한 단계씩 생각·행동 전체 계획을 먼저 수립 후 실행
장기 과제 도중에 큰 그림을 잃기 쉬움 명시적 계획으로 안정적 진행
계획 수정 매 스텝 즉흥적 단계 완료마다 명시적 재계획
모델 사용 보통 단일 모델 계획=강한 모델 / 실행=가벼운 모델 분리 가능

즉 장점은 둘입니다. ① 강력한 LLM도 어려워하는 장기 계획을 명시적으로 세워 안정성을 높이고, ② 계획엔 큰 모델, 실행엔 작은 모델을 써서 비용·자원을 최적화할 수 있습니다.

02전체 흐름 — 실행 ↔ 재계획 루프

가운데 세로줄이 기본 흐름입니다. replan이 매번 더 할 일이 남았는지 판단해, 남았으면 execute로 되돌아가고(재실행 루프), 끝났으면 보고서 작성 후 종료합니다.

START
planner
목표 → 전체 단계 리스트 생성
execute
계획의 첫 단계 1개를
ReAct 에이전트로 실행
replan
남은 계획 갱신 or 완료 판단
final_report
누적 결과로 보고서 작성
END
응답 준비됨
할 일 남음 → 재실행
계획 (planner · replan) 실행 (execute) 재실행 루프 완료 (보고서 · 종료) 시작 / 종료

03상태(State) — 계획과 결과를 함께 들고 다닌다

네 개의 값이 노드 사이를 흐릅니다. 특히 past_steps는 operator.add로 선언되어 실행할 때마다 결과가 지워지지 않고 누적되고, response가 채워지는 순간이 종료 신호입니다.

input str
사용자의 원래 질문·목표. 계획과 재계획의 기준점이 된다.
plan List[str]
현재 남은 계획 단계들. 재계획 때 통째로 새로 채워진다.
past_steps List[Tuple] · add
(단계, 실행 결과) 쌍을 계속 누적. 재계획·보고서의 근거가 된다.
response str
최종 응답. 값이 채워지면 곧 종료(→ 보고서) 신호로 쓰인다.

04노드 구성

execute가 계획 전체가 아니라 첫 단계 하나(plan[0])만 실행한다는 점이 포인트입니다. 그래야 한 단계 끝날 때마다 replan이 상황을 다시 볼 수 있습니다.

노드하는 일갱신하는 키
planner input을 보고 전체 단계 리스트를 만든다 plan
execute 계획의 첫 단계를 ReAct 에이전트로 실행 past_steps
replan 결과를 반영해 남은 계획을 갱신하거나, 끝이면 응답을 낸다 plan · response
final_report 누적된 past_steps로 마크다운 최종 보고서 작성 response

05계획과 재계획 — 두 개의 구조화 출력

LLM이 자유 문장을 뱉지 않도록 with_structured_output으로 정해진 형태만 내게 합니다. 계획은 단계 리스트, 재계획은 "끝낼지 / 계속할지"를 타입으로 표현하는 게 핵심입니다.

① Planner — 단계 리스트만 생성

Plan 모델은 steps 하나뿐. 목표를 순서 있는 개별 작업들로 쪼개게 합니다.

② Re-planner — Union으로 "종료 or 계속"을 결정

재계획기의 출력 타입이 Union[Response, Plan]입니다. LLM이 어떤 타입을 고르느냐가 곧 분기입니다 — Response를 고르면 "이제 답할 수 있음(종료)", Plan을 고르면 "아직 남았음(계속)".

Plan / Act 모델 · replan_step
# ── 계획: 실행할 단계 리스트 ──
class Plan(BaseModel):
    steps: List[str]                  # 순서대로 수행할 단계들

# ── 재계획: 끝낼지 / 계속할지 ──
class Response(BaseModel):
    response: str                     # 사용자에게 줄 최종 응답

class Act(BaseModel):
    action: Union[Response, Plan]     # Response=종료, Plan=계속

def replan_step(state):
    output = replanner.invoke(state)
    if isinstance(output.action, Response):
        return {"response": output.action.response}   # ← 종료 신호
    return {"plan": output.action.steps}             # ← 남은 계획 갱신

재계획 프롬프트는 "이미 끝낸 단계는 빼고, 아직 해야 할 단계만" 다시 나열하라고 지시합니다. 그래서 반복할수록 plan이 점점 짧아지다가 비게 되고, 결국 Response로 마무리됩니다.

06종료 판단 — should_end

replan 뒤에 붙는 조건부 엣지입니다. 상태에 response가 채워졌으면 보고서로, 아니면 다시 실행으로 보냅니다. 딱 이 한 가지로 루프의 지속/종료가 갈립니다.

should_end() · 루프 지속 여부
def should_end(state):
    # response가 채워졌으면 종료(보고서), 아니면 계속 실행
    if "response" in state and state["response"]:
        return "final_report"
    return "execute"

07실행 예시 — 한 질문이 흘러가는 모습

recursion_limit=50으로 넉넉히 두고 실행합니다. 계획이 단계마다 짧아지므로 보통은 몇 번의 execute ↔ replan 반복 뒤 자연스럽게 보고서로 끝납니다.

계획 → 단계별 실행 → 재계획 반복 → 최종 보고서 response → END

질문: "Modular RAG가 기존 Naive RAG와 어떤 차이가 있는지와 production level에서 쓰는 이점을 설명해줘"

① planner가 세운 계획 (예시)

  • 1.Naive RAG의 개념과 한계 조사
  • 2.Modular RAG의 개념과 구성 요소 조사
  • 3.두 방식의 핵심 차이점 비교
  • 4.production 환경에서의 이점 정리

② 실행 ↔ 재계획 루프

planner → execute (1) → replan ↺ → execute (2) → replan ↺ → ⋯ → final_report → END

매 execute는 남은 계획의 첫 단계 하나만 Tavily 검색 기반 ReAct 에이전트로 수행하고, 결과를 past_steps에 쌓습니다. replan은 그 결과를 보고 남은 단계를 다시 정리하다가, 더 할 일이 없어지면 Response를 반환 → response가 채워지고 → final_report가 누적 결과를 마크다운 보고서로 정리하며 종료합니다.

08그래프 조립 코드

대부분 일반 엣지로 이어지고, 조건부 엣지는 replan 뒤 한 곳뿐입니다. 이 한 갈래가 execute로 돌아가는 루프와 final_report로 나가는 출구를 동시에 만듭니다.

workflow 구성 · StateGraph
workflow.add_edge(START, "planner")
workflow.add_edge("planner", "execute")   # 계획 → 첫 단계 실행
workflow.add_edge("execute", "replan")    # 실행 → 재계획

# 재계획 후: 끝낼지 / 더 실행할지
workflow.add_conditional_edges(
    "replan", should_end,
    {"execute": "execute",          # 할 일 남음 → 재실행 루프
     "final_report": "final_report"}, # 응답 준비됨 → 보고서
)
workflow.add_edge("final_report", END)

app = workflow.compile(checkpointer=MemorySaver())

09핵심 정리

  • Plan-and-Execute는 계획을 먼저, 실행은 한 단계씩 하는 에이전트 전략이다.
  • ReAct의 즉흥적 진행과 달리 명시적 장기 계획으로 긴 작업에서도 길을 잃지 않는다.
  • execute는 계획 전체가 아니라 첫 단계 하나만 실행하고, 그 뒤 replan이 상황을 다시 본다.
  • 재계획기는 Union[Response, Plan]으로 "종료(Response) or 계속(Plan)"을 타입으로 결정한다.
  • response가 채워지는 순간이 종료 신호 — should_end가 이를 보고 보고서로 내보낸다.
  • 계획엔 강한 모델, 실행엔 가벼운 모델을 나눠 쓰면 비용·자원까지 최적화할 수 있다.
