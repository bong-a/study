여러 개의 Supervisor가 층을 이루는 계층적 에이전트 팀

Hierarchical Agent Team — 하나의 Supervisor가 모든 작업자를 직접 관리하는 대신, Research Team과 Doc Writing Team이라는 전문화된 하위 팀을 만들고, 상위 Supervisor가 두 팀에게 작업을 위임합니다. 각 팀은 독립적으로 컴파일된 LangGraph 그래프이며, 상위 그래프에서는 하나의 노드처럼 취급됩니다.

📓 08-LangGraph-Hierarchial-Agent-Team.ipynb
한눈에 보기

이전 Supervisor 예제는 Supervisor 1개 → Worker N개 구조였습니다. 작업자 수가 늘거나 서로 다른 전문 영역(웹 리서치 vs 문서 작성)이 섞이면, 하나의 Supervisor가 모든 걸 판단하기 벅차집니다.

이 노트북은 AutoGen 논문의 아이디어를 LangGraph로 구현하여, 이를 3단 계층으로 풀어냅니다.

사전 준비: load_dotenv()로 API 키 로드 → logging.langsmith(...)로 LangSmith 추적 설정 → MODEL_NAME = get_model_name(LLMs.GPT4o)로 사용할 모델 지정.

  • 1도구 생성 — 웹 리서치용 도구와 문서 작성용 도구를 각각 정의한다.
  • 2팀 정의 — Research Team, Doc Writing Team을 각각 독립된 서브그래프로 만든다.
  • 3계층 추가 — 상위 Supervisor가 두 팀을 라우팅하는 Super-Graph를 만든다.
  • 4결합 — 서브그래프를 파이프(|)로 감싸 상위 그래프의 노드로 등록한다.

01왜 계층적 구조가 필요한가

단일 Supervisor 방식이 간단하고 효율적인 경우도 많지만, 다음 두 가지 상황에서는 계층(Hierarchy)을 도입하는 편이 유리합니다.

항목 단일 Supervisor 계층적 Supervisor
관리 대상 모든 Worker를 한 곳에서 직접 라우팅 상위는 팀 단위로, 팀 내부는 팀 Supervisor가 라우팅
작업 복잡성 서로 다른 전문 영역이 섞이면 판단이 어려워짐 영역별 팀(Research / Doc Writing)으로 분리해 전문화
작업자 수 확장 Worker가 늘수록 라우팅 옵션·프롬프트가 비대해짐 팀 내부에만 Worker를 추가하면 되어 국소적으로 확장
그래프 구조 단일 StateGraph 팀별 서브그래프 + 이를 감싸는 Super-Graph

정확히는 상위 Supervisor가 "ResearchTeam" / "PaperWritingTeam"이라는 팀 노드에 작업을 위임하고, 그 팀 노드 내부의 sub-supervisor가 다시 소속 Worker에게 작업을 재할당하는 구조입니다.

02전체 아키텍처 — 3단 계층

점선 박스는 독립적으로 컴파일된 서브그래프입니다. 상위 Supervisor 입장에서는 내부 구조를 몰라도 되고, "ResearchTeam", "PaperWritingTeam"이라는 노드 하나씩으로만 보입니다.

START

ResearchTeam (서브그래프)

화살표는 이해를 돕기 위해 양방향으로 표시했지만, 실제로는 방향이 다릅니다: Supervisor → Worker/팀은 add_conditional_edges(라우팅 결정에 따른 조건부 이동)이고, Worker/팀 → Supervisor는 add_edge(항상 되돌아오는 고정 엣지)입니다.

03팀별 도구 구성

각 팀의 Worker는 팀 목적에 맞는 도구만 갖습니다. Research Team은 웹에서 정보를 가져오는 도구를, Doc Writing Team은 로컬 파일 시스템에 읽고 쓰는 도구를 사용합니다.

팀도구역할
Research tavily_tool TavilySearch(max_results=5) — 웹 검색 엔진 도구
scrape_webpages WebBaseLoader로 URL 목록을 로드해 <Document name=...> 형식 문자열로 합침
Doc Writing create_outline 주요 포인트 리스트를 받아 아웃라인 파일로 저장
read_document 파일 내용을 읽어 반환 (end까지 자르는 로직은 있으나, start가 주어져도 코드상 다시 0으로 재설정되는 점에 유의)
write_document 새 텍스트 문서를 작성·저장
edit_document 줄 번호 → 삽입 텍스트 딕셔너리로 기존 문서에 내용 삽입
python_repl_tool PythonREPLTool — 차트 등 필요한 코드를 직접 실행

⚠️ python_repl_tool과 파일 접근 도구는 에이전트가 로컬 환경에 직접 쓰기 작업을 하므로, 실제 프로덕션에서는 샌드박싱 등 안전장치가 필요합니다.

04공통 유틸리티 — 에이전트/Supervisor를 찍어내는 팩토리

팀마다 Worker와 Supervisor를 반복해서 만들어야 하므로, 이를 함수/클래스로 추상화합니다. 핵심은 Supervisor의 구조화 출력입니다 — 자유 문장이 아니라 "다음 작업자 이름" 또는 "FINISH" 중 하나만 뱉도록 타입으로 강제합니다.

AgentFactory · create_team_supervisor
# ── Worker 노드로 변환: agent 실행 결과를 HumanMessage(name=팀원명)로 감싼다 ──
class AgentFactory:
    def create_agent_node(self, agent, name: str):
        def agent_node(state):
            result = agent.invoke(state)
            return {"messages": [HumanMessage(content=result["messages"][-1].content, name=name)]}
        return agent_node

# ── Supervisor: 멤버 목록으로 Literal 타입을 동적으로 만들어 구조화 출력 강제 ──
def create_team_supervisor(model_name, system_prompt, members):
    options_for_next = ["FINISH"] + members

    class RouteResponse(BaseModel):
        next: Literal[*options_for_next]   # ← "FINISH" 또는 멤버 이름 중 하나만 허용

    prompt = ChatPromptTemplate.from_messages([
        ("system", system_prompt),
        MessagesPlaceholder(variable_name="messages"),
        ("system", "who should act next? Select one of: {options}"),
    ]).partial(options=str(options_for_next))

    return prompt | ChatOpenAI(model=model_name, temperature=0).with_structured_output(RouteResponse)

Literal[*options_for_next]가 핵심 트릭입니다. 팀마다 멤버 구성이 다르므로, 런타임에 멤버 리스트를 받아 Literal 타입을 즉석에서 조립합니다. 덕분에 LLM은 실제로 존재하는 팀원 이름 중 하나 또는 "FINISH"만 고를 수 있습니다. (참고: 노트북 코드의 타입 힌트는 -> str이지만, 실제 반환값은 문자열이 아니라 prompt | llm.with_structured_output(...)로 만들어진 Runnable 체인입니다.)

05Research Team — 상태·노드·그래프

Searcher와 WebScraper 두 Worker를 Supervisor가 조율하는, 가장 단순한 형태의 서브그래프입니다.

messages List[BaseMessage] · add
지금까지의 대화(작업 결과) 누적. Worker가 결과를 낼 때마다 쌓인다.
team_members List[str]
이 팀에 속한 Worker 이름 목록.
next str
Supervisor가 정한 "다음 실행할 노드" — 조건부 엣지가 이 값을 보고 라우팅한다.

실행 흐름 예시 (가능한 한 경로)

아래는 Supervisor가 선택할 수 있는 여러 경로 중 하나입니다. 실제로는 매 턴 Supervisor의 구조화 출력(next)에 따라 Searcher/WebScraper 중 무엇을, 몇 번 반복할지가 달라집니다.

Supervisor → Searcher → Supervisor ↺ → WebScraper → Supervisor → FINISH

get_next_node(x): return x["next"] 하나로 조건부 엣지를 구성합니다. Supervisor가 next="FINISH"를 반환하면 END로, 그 외에는 Supervisor가 지목한 Worker로 이동합니다 — Worker끼리 직접 연결된 엣지는 없습니다. 그래프는 MemorySaver() 체크포인터와 함께 컴파일되어(compile(checkpointer=MemorySaver())), thread_id 단위로 대화 상태가 유지됩니다.

⚠️ 노트북의 Supervisor 시스템 프롬프트 문자열은 워커를 "Search"로 표기하지만, 실제 라우팅 옵션과 노드 이름은 "Searcher"입니다. 코드를 그대로 복사해 쓸 때 이름을 맞춰주세요.

06Doc Writing Team — 상태·노드·그래프

3개의 Worker(NoteTaker, DocWriter, ChartGenerator)가 파일 시스템을 공유합니다. Research Team과 달리, 각 Worker 앞에 파이프로 합성된 preprocess 함수가 매번 현재 작업 디렉토리 상태를 에이전트 입력에 얹어준다는 점이 특징입니다 (그래프에 추가되는 별도의 노드가 아니라, Worker Runnable 자체를 감싸는 전처리 단계입니다).

messages List[BaseMessage] · add
대화(작업 결과) 누적.
team_members str
이 팀에 속한 Worker 이름 목록(문자열).
next str
Supervisor가 정한 "다음 실행할 노드" — 조건부 엣지가 이 값을 보고 라우팅한다.
current_files str
preprocess가 ./tmp 디렉토리를 스캔해 채워주는, "지금까지 작성된 파일 목록".
preprocess · Worker 앞에 파이프로 합성되는 전처리 함수
def preprocess(state):
    written_files = [f.relative_to(WORKING_DIRECTORY) for f in WORKING_DIRECTORY.rglob("*")]
    if not written_files:
        return {**state, "current_files": "No files written."}
    return {**state, "current_files": "Below are files your team has written:\n" + "\n".join(f" - {f}" for f in written_files)}

# 파이프로 연결: 매 호출마다 preprocess가 먼저 실행되어 agent 입력에 현재 파일 목록을 얹는다
context_aware_doc_writer_agent = preprocess | doc_writer_agent
노드보유 도구역할
NoteTaker create_outline, read_document 주제에 대한 아웃라인(목차)을 먼저 작성
DocWriter write_document, edit_document, read_document 아웃라인을 바탕으로 논문 스타일 본문 작성
ChartGenerator read_document, python_repl_tool 필요 시 문서 내용을 바탕으로 차트 코드 실행

실행 흐름 예시 (가능한 한 경로)

이 역시 Supervisor가 매 턴 next로 세 Worker 중 하나를 반복 선택한 결과이며, 아래는 자연스러운 순서 중 하나(아웃라인 → 본문 → 차트)를 예시로 보여줍니다.

Supervisor → NoteTaker → Supervisor ↺ → DocWriter → Supervisor ↺ → ChartGenerator → Supervisor → FINISH

이 그래프도 MemorySaver() 체크포인터와 함께 컴파일됩니다 (authoring_graph.compile(checkpointer=MemorySaver())).

07Super-Graph — 두 팀을 하나로 묶기

상위 그래프의 상태는 아주 단순합니다(messages, next뿐). 핵심은 이미 컴파일된 서브그래프(web_research_app, authoring_app)를 파이프로 감싸 그대로 노드로 등록한다는 점입니다.

get_last_message · join_graph · Super-Graph 조립
# 상위 그래프 진입 시: 마지막 메시지 하나만 서브그래프 입력으로 변환
def get_last_message(state: State) -> str:
    last_message = state["messages"][-1]
    if isinstance(last_message, str):
        return {"messages": [HumanMessage(content=last_message)]}
    else:
        return {"messages": [last_message.content]}   # 메시지 객체면 content(str)만 그대로 담음

# 서브그래프 실행 후: 마지막 메시지만 뽑아 상위 상태로 되돌림
def join_graph(response: dict):
    return {"messages": [response["messages"][-1]]}

# ── 서브그래프를 파이프로 감싸 "하나의 노드"처럼 등록 ──
super_graph.add_node("ResearchTeam", get_last_message | web_research_app | join_graph)
super_graph.add_node("PaperWritingTeam", get_last_message | authoring_app | join_graph)
super_graph.add_node("Supervisor", supervisor_node)

super_graph.add_conditional_edges(
    "Supervisor", get_next_node,
    {"PaperWritingTeam": "PaperWritingTeam", "ResearchTeam": "ResearchTeam", "FINISH": END},
)
super_graph.set_entry_point("Supervisor")

즉 get_last_message | web_research_app | join_graph 자체가 하나의 Runnable이고, LangGraph 입장에서는 이것이 그냥 "ResearchTeam" 노드 하나입니다. 서브그래프 내부에 몇 개의 Worker와 루프가 있는지는 상위 Supervisor가 전혀 알 필요가 없습니다 — 계층적 캡슐화의 핵심입니다. Super-Graph 역시 MemorySaver() 체크포인터와 함께 컴파일됩니다.

08실행 예시 — 논문 작성 요청이 흘러가는 모습

"멀티 에이전트 구조에 대한 arXiv 스타일 리포트를 작성해줘"라는 요청이 상위 Supervisor를 거쳐 두 팀을 오가는 대표적인 한 경로를 봅니다. 실제 순서는 매 턴 상위 Supervisor의 구조화 출력(next)에 따라 달라질 수 있습니다. (recursion_limit=150으로 넉넉하게 설정)

Supervisor → ResearchTeam ↺ → Supervisor → PaperWritingTeam ↺ → Supervisor → FINISH next → END

요청: "multi-agent 구조를 사용하여 복잡한 작업을 수행하는 방법에 대한 Arxiv 논문 형식 리포트를 한글로 작성 — 아웃라인, 각 섹션 5문장 이상, 필요시 차트, APA 출처, .md 저장"

① 최상위 Supervisor의 라우팅 흐름 (예시)

Supervisor → ResearchTeam (내부: Searcher/WebScraper 중 Supervisor가 반복 선택) → Supervisor ↺ → PaperWritingTeam (내부: NoteTaker/DocWriter/ChartGenerator 중 Supervisor가 반복 선택) → Supervisor → FINISH

상위 Supervisor는 먼저 ResearchTeam에게 위임해 자료를 수집시키고, 결과가 돌아오면 PaperWritingTeam에게 넘겨 문서를 완성시키는 흐름이 자연스럽습니다. 각 팀 내부에서 Supervisor가 어떤 Worker를 몇 번 반복 선택하는지는 상위 Supervisor에게 보이지 않으며, 팀이 FINISH를 반환한 뒤의 최종 메시지만 상위로 전달됩니다.

09핵심 정리

  • 단일 Supervisor로 벅찬 작업은 팀 단위로 쪼개고, 상위 Supervisor가 팀 간 라우팅만 맡는다.
  • 각 팀(Research / Doc Writing)은 독립적으로 컴파일된 서브그래프이며, 자체 State·Worker·Supervisor를 가진다.
  • Supervisor는 Literal[*options_for_next]로 멤버 이름 또는 FINISH만 고르도록 구조화 출력을 강제한다.
  • Doc Writing Team은 preprocess 노드로 현재 작업 디렉토리 상태를 매번 에이전트에 주입한다.
  • 서브그래프는 get_last_message | subgraph | join_graph 파이프로 감싸면 상위 그래프의 노드 하나가 된다.
  • 이 캡슐화 덕분에 상위 Supervisor는 팀 내부의 루프·Worker 수를 몰라도 되고, 팀을 추가·교체하기도 쉬워진다.
