검색 결과를 스스로 채점하고, 부족하면 웹에서 보완하는 RAG

CRAG (Corrective RAG) — 검색해 온 문서가 질문에 정말 맞는지 LLM이 직접 채점하고, 쓸 만한 문서가 하나도 없으면 질문을 고쳐 웹 검색으로 보완한 뒤 답을 만드는 자기 교정형(self-corrective) RAG 파이프라인입니다.

일반적인 RAG(Naive RAG)는 검색된 문서를 그대로 믿고 바로 답을 만듭니다. 문서가 엉뚱해도 그 위에서 답을 지어내니 환각(hallucination) 위험이 있죠.

CRAG는 답을 만들기 전에 한 단계를 더 끼워 넣습니다 — 검색된 각 문서가 질문과 관련 있는지 yes/no로 채점하고, 관련 문서가 하나도 없으면 질문을 다듬어 웹 검색으로 부족한 지식을 보강합니다. 이 "검수 → 보정" 루프가 CRAG의 핵심입니다.

01Naive RAG vs Corrective RAG

두 방식의 가장 큰 차이는 "검색 결과를 얼마나 믿느냐"입니다. CRAG는 검색 결과를 일단 의심하고 검수합니다.

비교 항목Naive RAGCorrective RAG
검색 문서 신뢰 검색된 문서를 그대로 사용 관련성을 채점해 선별 후 사용
관련 문서가 없을 때 그대로 생성 (환각 위험) 웹 검색으로 지식 보강
질문 처리 원본 질문 그대로 검색 필요 시 질문 재작성 후 검색
자기 점검 없음 self-evaluation 단계 포함

원 논문(Corrective Retrieval Augmented Generation)은 문서를 "knowledge strips"로 잘게 나눠 정제하는 단계까지 제안하지만, 이 튜토리얼은 지식 정제 단계는 생략하고 "관련 문서가 0개면 웹 검색으로 보강"이라는 핵심 아이디어만 LangGraph로 구현합니다.

03파이프라인 4가지 부품

그래프의 노드들은 아래 4개의 LangChain 체인·도구를 각각 감싼 것입니다.

Retrieval Grader

검색된 문서 한 개씩 질문과의 관련성을 yes/no 이진 점수로 채점. 구조화 출력(GradeDocuments)으로 형식을 강제.

with_structured_output(GradeDocuments)
Generate

선별된 문서(+웹 결과)를 컨텍스트로 답을 만드는 일반 RAG 체인. Hub의 teddynote/rag-prompt 사용.

prompt | llm | StrOutputParser()
Question Re-writer

원본 질문의 숨은 의도를 파악해 웹 검색에 더 잘 맞는 형태로 다시 씀. 보정 경로에서만 동작.

re_write_prompt | llm | StrOutputParser()
Web Search Tool

부족한 지식을 외부에서 채우는 Tavily 검색(최대 3건). 결과를 Document로 묶어 문서 목록에 추가.

TavilySearch(max_results=3)

04상태(State) — 노드끼리 주고받는 값

모든 노드는 이 GraphState 딕셔너리를 읽고 일부 키만 갱신해 반환합니다. 특히 web_search가 경로를 가르는 스위치 역할을 합니다.

question str
사용자 질문. 재작성되면 이 값이 교체된다.
generation str
LLM이 만든 최종 답변.
web_search str
"Yes"/"No" — 웹 검색이 필요한지. 분기의 기준.
documents List[str]
검색·선별된 문서 목록. 웹 결과도 여기에 덧붙는다.

06분기의 핵심 — decide_to_generate

grade_documents 뒤에 붙는 조건부 엣지. 채점이 정한 web_search 값만 보고 다음 노드를 고릅니다. 규칙은 단 두 줄입니다.

web_search 값이동의미
"No" generate 관련 문서가 하나라도 있음 → 바로 답변 생성 (정상 경로)
"Yes" query_rewrite 관련 문서가 0개 → 질문 재작성 후 웹 검색으로 보정
decide_to_generate() · 조건부 라우팅
def decide_to_generate(state):
    if state["web_search"] == "Yes":
        # 관련 문서 없음 → 보정 경로
        return "query_rewrite"
    else:
        # 관련 문서 있음 → 바로 생성
        return "generate"

07실행 예시 — 두 경로 비교

같은 그래프라도 질문에 따라 다른 길을 탑니다. 실습 PDF는 SPRi AI Brief 2023년 12월호라는 점이 열쇠입니다.

정상 경로 — 문서에 답이 있는 경우 web_search = No

질문: "삼성전자가 개발한 생성형 AI 의 이름은?"

retrieve → grade_documents → generate → END

PDF 안에 관련 내용이 있어 채점을 통과한 문서가 남습니다. 웹 검색 없이 곧바로 생성 단계로 넘어가 "삼성 가우스"라는 답을 만들어 냅니다.

보정 경로 — 문서에 답이 없는 경우 web_search = Yes

질문: "2024년 노벨문학상 수상자의 이름은?"

retrieve → grade_documents → query_rewrite → web_search_node → generate → END

PDF는 2023년 자료라 노벨문학상 정보가 없습니다. 모든 문서가 채점에서 탈락(관련 문서 0개)해 web_search="Yes"가 되고, 질문을 재작성한 뒤 웹 검색으로 최신 정보를 보강해 답을 만듭니다.

08그래프 조립 코드

노드와 엣지를 연결하면 앞의 다이어그램이 그대로 코드가 됩니다. 핵심은 grade_documents에 건 조건부 엣지 하나입니다.

workflow 구성 · StateGraph
workflow.add_edge(START, "retrieve")
workflow.add_edge("retrieve", "grade_documents")

# 채점 결과에 따라 두 갈래로 분기
workflow.add_conditional_edges(
    "grade_documents",
    decide_to_generate,
    {
        "query_rewrite": "query_rewrite",  # 보정 경로
        "generate": "generate",            # 정상 경로
    },
)

# 보정 경로: 재작성 → 웹 검색 → 생성으로 합류
workflow.add_edge("query_rewrite", "web_search_node")
workflow.add_edge("web_search_node", "generate")
workflow.add_edge("generate", END)

app = workflow.compile()

09핵심 정리

  • CRAG는 답을 만들기 전에 검색 결과를 채점하는 "검수 단계"를 추가한 RAG다.
  • 채점은 문서 하나씩 yes/no 이진 점수로, 관련된 것만 남긴다.
  • 관련 문서가 0개면 web_search="Yes" → 질문 재작성 → 웹 검색으로 보강한다.
  • 분기 decide_to_generate는 오직 web_search 값 하나로 경로를 정한다.
  • 정상 경로와 보정 경로는 모두 generate에서 합류해 최종 답을 만든다.
  • 이 튜토리얼은 원 논문의 지식 정제 단계는 생략한 간소화 버전이다.
