세 줄 정리

① 문제

원시 LLM에 한 번 프롬프트를 던져 만든 보고서는 얕고 일반적인 내용에 그치기 쉽다. 다양한 관점과 충분한 근거 조사 없이는 실제 의사결정 워크플로우에 쓰기 어렵다.

② 해법

연구 주제를 하위 관점으로 쪼개 AI 분석가 팀을 만들고(Human-in-the-loop로 팀 구성 검토), 각 분석가가 전문가 역할 LLM과 병렬로 인터뷰하며 웹·Arxiv에서 근거를 수집한다.

③ 효과

모든 인터뷰 결과가 map-reduce로 병합되어 출처가 인용된 서론-본문-결론 보고서가 자동 완성된다. max_analysts만 늘리면 조사 폭이 그대로 확장된다.

00 · 전체 흐름

리포트를 보고 인터뷰하는 것이 아니라, 인터뷰와 검색으로 리포트를 만든다

최종 보고서는 인터뷰 이후의 산출물이다. 각 분석가의 질문이 검색 쿼리로 바뀌고, 웹·Arxiv에서 가져온 문서가 전문가 답변의 근거가 되며, 그 답변과 소스 문서를 섹션 메모로 압축한 뒤 마지막에 하나의 보고서로 합친다.

핵심은 방향이다: 리포트 → 인터뷰가 아니라, 질문 → 웹/Arxiv 검색 → context 기반 답변 → 섹션 메모 → 최종 리포트다.

01 · 배경

STORM — 관점 기반 대화 시뮬레이션으로 리서치를 자동화

연구는 종종 분석가에게 위임되는 노동 집약적인 작업이다. 그러나 원시 LLM 출력은 실제 의사 결정 워크플로우에 적합하지 않은 경우가 많다. STORM 논문은 이 문제를 두 가지 아이디어로 해결한다.

유사 주제 조회

관련된 다른 주제들을 조사해, 질문할 만한 흥미로운 관점을 넓힌다.

다양한 관점의 대화 시뮬레이션

서로 다른 관점을 가진 여러 화자가 주제를 놓고 대화하며, 참고 출처 사용 빈도와 정보 밀도를 함께 끌어올린다.

02 · 분석가 생성 (Human-in-the-loop)

Analyst 페르소나 팀을 만들고 사람이 검토한다

연구의 첫 단계는 주제를 여러 하위 관점으로 나누는 것이다. Perspectives 스키마로 LLM에게 구조화된 출력을 강제해, 서로 다른 소속·역할을 가진 분석가 목록을 만든다.

class Analyst(BaseModel):
    affiliation: str = Field(description="Primary affiliation of the analyst.")
    name: str = Field(description="Name of the analyst.")
    role: str = Field(description="Role of the analyst in the context of the topic.")
    description: str = Field(description="Description of the analyst focus, concerns, and motives.")

    @property
    def persona(self) -> str:
        return f"Name: {self.name}\nRole: {self.role}\nAffiliation: {self.affiliation}\nDescription: {self.description}\n"

class Perspectives(BaseModel):
    analysts: List[Analyst] = Field(description="Comprehensive list of analysts with their roles and affiliations.")

def create_analysts(state: GenerateAnalystsState):
    topic = state["topic"]
    max_analysts = state["max_analysts"]
    human_analyst_feedback = state.get("human_analyst_feedback", "")

    structured_llm = llm.with_structured_output(Perspectives)
    system_message = analyst_instructions.format(
        topic=topic, human_analyst_feedback=human_analyst_feedback, max_analysts=max_analysts
    )
    analysts = structured_llm.invoke(
        [SystemMessage(content=system_message)] + [HumanMessage(content="Generate the set of analysts.")]
    )
    return {"analysts": analysts.analysts}

def human_feedback(state: GenerateAnalystsState):
    """사용자 피드백을 받기 위한 중단점 노드 — 본문은 비워 둔다"""
    pass

def should_continue(state: GenerateAnalystsState):
    human_analyst_feedback = state.get("human_analyst_feedback", None)
    if human_analyst_feedback:
        return "create_analysts"
    return END
human_feedback 노드가 비어 있는 이유

human_feedback는 실행 로직이 없는 중단점 전용 노드다. 그래프를 interrupt_before=["human_feedback"]로 컴파일하면, LangGraph는 이 노드를 실행하기 직전에 멈춘다. 이때 graph.update_state(..., as_node="human_feedback")로 사람이 대신 상태를 채워 넣고, should_continue가 피드백 유무에 따라 재생성 또는 종료를 결정한다.

분석가 생성 그래프 — Human-in-the-loop 루프

실선(회색/빨강) = 고정 엣지 · 점선(indigo) = should_continue 조건부 엣지 · 피드백이 있는 한 create_analysts ↔ human_feedback 루프가 반복된다.

그래프 구성 코드

builder = StateGraph(GenerateAnalystsState)
builder.add_node("create_analysts", create_analysts)
builder.add_node("human_feedback", human_feedback)

builder.add_edge(START, "create_analysts")
builder.add_edge("create_analysts", "human_feedback")
builder.add_conditional_edges("human_feedback", should_continue, ["create_analysts", END])

memory = MemorySaver()
graph = builder.compile(interrupt_before=["human_feedback"], checkpointer=memory)

단독 실행 예시 — 피드백을 두 번 주고받기

이 데모는 topic과 max_analysts=3만 넣고 실행한 뒤, 두 차례에 걸쳐 human_feedback 노드의 역할을 사람이 대신 수행한다. 노트북에는 실제 LLM 호출 결과가 저장되어 있지 않으므로, 아래는 코드가 만드는 제어 흐름이다.

1
invoke_graph(graph, inputs, config)

첫 인터럽트까지 실행

topic="Modular RAG 가 기존의 Naive RAG 와 어떤 차이가 있는지...", max_analysts=3으로 실행하면 create_analysts가 끝난 뒤 human_feedback 직전에 __interrupt__가 출력되며 멈춘다.

2
graph.get_state(config).next

다음 실행 노드 확인

state.next가 ('human_feedback',)로 나오면, 사람이 피드백을 넣을 준비가 된 것이다.

3
graph.update_state(..., as_node="human_feedback")

피드백 주입

human_analyst_feedback="Add in someone named Teddy Lee from a startup to add an entrepreneur perspective"를 넣고, as_node="human_feedback"로 그 노드가 방금 이 값을 반환한 것처럼 상태를 갱신한다.

4
invoke_graph(graph, None, config)

재개 → 재생성 → 다시 인터럽트

입력에 None을 주면 그래프가 이어서 진행된다. should_continue가 피드백이 있음을 확인해 create_analysts로 돌아가 분석가를 다시 만들고, 다시 human_feedback에서 멈춘다.

5
human_analyst_feedback = None → 재개

추가 피드백 없이 종료

이번에는 human_analyst_feedback에 None을 넣고 재개하면 should_continue가 END를 반환한다. final_state.next는 빈 튜플 ()이 되어 모든 작업이 끝났음을 나타낸다.

config = RunnableConfig(recursion_limit=10, configurable={"thread_id": random_uuid()})
inputs = {"topic": topic, "max_analysts": 3}
invoke_graph(graph, inputs, config)                      # → __interrupt__ 로 정지

graph.update_state(
    config,
    {"human_analyst_feedback": "Add in someone named Teddy Lee from a startup to add an entrepreneur perspective"},
    as_node="human_feedback",
)
invoke_graph(graph, None, config)                          # 재생성 후 다시 정지

graph.update_state(config, {"human_analyst_feedback": None}, as_node="human_feedback")
invoke_graph(graph, None, config)                          # 종료 (END)

※ 재개 시 입력을 None으로 주는 것은 "새 값 없이 중단된 지점부터 계속 진행하라"는 신호다.

도구 — 웹 검색과 Arxiv 검색을 병렬로

전문가는 질문에 답하기 위해 여러 소스에서 정보를 동시에 수집한다. 이 튜토리얼은 Tavily(웹)와 Arxiv(논문) 두 가지를 사용한다.

노드도구파라미터비고
search_webTavilySearchmax_results=3대화의 마지막 질문을 검색 쿼리로 변환 후 웹 검색
search_arxivArxivRetrieverload_max_docs=2 (호출 시 재지정)⚠ 초기화 시 기본값은 load_max_docs=3이지만, 노드 내부 호출에서 2로 다시 지정한다
search_arxiv 의 예외 처리

Arxiv 검색은 외부 API 호출이라 실패할 수 있다. search_arxiv는 try/except로 감싸, 실패 시 <Error>Arxiv 검색 결과를 가져오는데 실패했습니다.</Error>를 context에 넣어 그래프 전체가 멈추지 않도록 한다.

def search_web(state: InterviewState):
    structured_llm = llm.with_structured_output(SearchQuery)
    search_query = structured_llm.invoke([search_instructions] + state["messages"])
    search_docs = tavily_search.invoke(search_query.search_query)
    formatted_search_docs = "\n\n---\n\n".join(
        [f'<Document href="{doc["url"]}"/>\n{doc["content"]}\n</Document>' for doc in search_docs]
    )
    return {"context": [formatted_search_docs]}

def search_arxiv(state: InterviewState):
    structured_llm = llm.with_structured_output(SearchQuery)
    search_query = structured_llm.invoke([search_instructions] + state["messages"])
    try:
        arxiv_search_results = arxiv_retriever.invoke(
            search_query.search_query, load_max_docs=2,
            load_all_available_meta=True, get_full_documents=True,
        )
        formatted_search_docs = "\n\n---\n\n".join([
            f'<Document source="{d.metadata["entry_id"]}" ...>\n<Title>{d.metadata["Title"]}</Title>\n'
            f'<Summary>{d.metadata["Summary"]}</Summary>\n<Content>{d.page_content}</Content>\n</Document>'
            for d in arxiv_search_results
        ])
        return {"context": [formatted_search_docs]}
    except Exception as e:
        return {"context": ["<Error>Arxiv 검색 결과를 가져오는데 실패했습니다.</Error>"]}

generate_answer — 인용을 포함한 답변

answer_instructions는 전문가 역할의 LLM에게 엄격한 인용 규칙을 지시한다.

  • context에 제공된 정보만 사용하고, 외부 지식이나 추측을 도입하지 않는다
  • 관련 문장 옆에 [1], [2]처럼 출처 번호를 표기한다
  • 답변 하단에 [1] 출처1, [2] 출처2 순서로 출처 목록을 정리한다
  • 같은 출처는 중복 없이 하나로 합친다

route_messages — 인터뷰를 언제 끝낼지 판단

Send() 로 모든 분석가의 인터뷰를 동시에 시작 (map)

def initiate_all_interviews(state: ResearchGraphState):
    human_analyst_feedback = state.get("human_analyst_feedback")
    if human_analyst_feedback:
        return "create_analysts"
    else:
        topic = state["topic"]
        return [
            Send(
                "conduct_interview",
                {"analyst": analyst, "messages": [HumanMessage(content=f"So you said you were writing an article on {topic}?")]},
            )
            for analyst in state["analysts"]
        ]

def write_report(state: ResearchGraphState):
    sections = state["sections"]
    formatted_str_sections = "\n\n".join([f"{section}" for section in sections])
    system_message = report_writer_instructions.format(topic=state["topic"], context=formatted_str_sections)
    report = llm.invoke([SystemMessage(content=system_message)] + [HumanMessage(content="Write a report based upon these memos.")])
    return {"content": report.content}

def finalize_report(state: ResearchGraphState):
    content = state["content"]
    if content.startswith("## Insights"):
        content = content.strip("## Insights")
    if "## Sources" in content:
        content, sources = content.split("\n## Sources\n")
    else:
        sources = None

    final_report = (
        state["introduction"] + "\n\n---\n\n## Main Idea\n\n" + content + "\n\n---\n\n" + state["conclusion"]
    )
    if sources is not None:
        final_report += "\n\n## Sources\n" + sources
    return {"final_report": final_report}
06 · 전체 그래프 조립

분석가 생성부터 최종 보고서까지 — 하나의 그래프로

지금까지 만든 모든 노드와 인터뷰 서브그래프를 하나의 ResearchGraphState 그래프에 연결한다.

엣지 종류연결결정 방식
시작 엣지START → create_analysts → human_feedback고정
조건부 엣지 (map)human_feedback → {create_analysts | conduct_interview × N}initiate_all_interviews — 피드백 있으면 재생성, 없으면 Send()
팬아웃 엣지conduct_interview → {write_report, write_introduction, write_conclusion}인터뷰 완료 후 세 작성 작업이 병렬 실행
조인 엣지 (reduce)[write_conclusion, write_report, write_introduction] → finalize_report세 노드가 모두 끝나야 실행 (barrier)
종료 엣지finalize_report → END고정

teal 실선 = map 단계(Send) 및 팬아웃 · indigo 점선 = 조건부/조인 엣지 · 빨간선 = 종료. finalize_report는 세 작성 노드가 모두 끝나야 실행되는 barrier다.

builder = StateGraph(ResearchGraphState)
builder.add_node("create_analysts", create_analysts)
builder.add_node("human_feedback", human_feedback)
builder.add_node("conduct_interview", interview_builder.compile())
builder.add_node("write_report", write_report)
builder.add_node("write_introduction", write_introduction)
builder.add_node("write_conclusion", write_conclusion)
builder.add_node("finalize_report", finalize_report)

builder.add_edge(START, "create_analysts")
builder.add_edge("create_analysts", "human_feedback")
builder.add_conditional_edges("human_feedback", initiate_all_interviews, ["create_analysts", "conduct_interview"])

builder.add_edge("conduct_interview", "write_report")
builder.add_edge("conduct_interview", "write_introduction")
builder.add_edge("conduct_interview", "write_conclusion")
builder.add_edge(["write_conclusion", "write_report", "write_introduction"], "finalize_report")
builder.add_edge("finalize_report", END)

memory = MemorySaver()
graph = builder.compile(interrupt_before=["human_feedback"], checkpointer=memory)

※ add_edge([...], "finalize_report")처럼 소스를 리스트로 주면, 나열된 노드가 모두 끝난 뒤에야 대상 노드가 실행되는 조인(barrier) 엣지가 된다.

1

08 · 비교 분석

단일 프롬프트 LLM vs STORM 멀티 에이전트

같은 주제를 두고 "LLM에게 한 번에 써 달라고 요청"하는 방식과, 이 튜토리얼의 분석가 팀 + 병렬 인터뷰 방식을 나란히 비교하면 차이가 분명해진다.

항목 단일 프롬프트 LLM STORM 멀티 에이전트 (이 튜토리얼)
관점 다양성 ✕ 하나의 시각으로만 서술 ✓ 서로 다른 소속·역할의 분석가 여럿
근거 수집 ✕ 모델 내부 지식에 의존, 환각 위험 ✓ 웹(Tavily) + 논문(Arxiv) 실시간 검색으로 근거 확보
사람의 개입 ✕ 한 번에 생성, 중간 수정 불가 ✓ interrupt_before로 분석가 팀 구성을 검토·수정
출력 구조 ~ 형식·일관성이 매번 달라짐 ✓ 서론 - 본문(섹션별 인용) - 결론 구조 고정
처리 비용·지연 ✓ LLM 호출 1회, 빠르고 저렴 ✕ 분석가 수 × 인터뷰 턴 수만큼 호출 필요 (단, 병렬 실행으로 지연은 완화)
확장성 ~ 주제가 복잡할수록 한계 ✓ max_analysts만 늘리면 조사 폭이 확장
09 · 정리

정리 · 언제 쓰나 · 유의점

✓ 이럴 때 쓰면 좋다

  • 주제를 여러 하위 관점으로 자연스럽게 쪼갤 수 있는 리서치
  • 인용·출처가 중요한 보고서를 작성해야 할 때
  • 분석가 팀 구성을 사람이 검토하고 조정하고 싶을 때
  • 웹과 논문 등 복수 소스를 병렬로 조사해야 할 때

⚠ 한계 · 유의점

  • 분석가 수 × 인터뷰 턴 수만큼 LLM 호출 비용이 발생한다
  • Tavily·Arxiv 등 외부 API 의존 — search_arxiv처럼 실패 처리가 필요하다
  • interrupt_before 구조상 완전 자동화는 아니며, 사람 개입 지점이 필요하다
  • 병렬 인터뷰의 기본 종료 턴(2턴)이 짧아, 필요하면 Send() 페이로드에 max_num_turns를 명시적으로 추가하는 것이 좋다
한 줄 정리

STORM 패턴은 "누가 무엇을 물어볼 것인가"를 사람과 LLM이 함께 설계하고, "정보 수집"은 병렬(map)로 흩어졌다가 "글쓰기"에서 다시 하나(reduce)로 모이는 구조로 연구 자동화를 구현한다.
