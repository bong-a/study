SQL 데이터베이스와 대화하며 스스로 오류를 고치는 에이전트

질문만 던지면 테이블을 찾고 → 스키마를 확인하고 → SQL을 작성·검수·실행한 뒤 답을 만드는 LangGraph 에이전트입니다. 쿼리가 틀리면 에러 메시지를 스스로 읽고 다시 쓰는 재시도 루프가 핵심이며, LangSmith로 답변 정확도까지 평가합니다.

한눈에 보기

이 에이전트는 chinook(디지털 음원 스토어) SQLite 데이터베이스를 대상으로, 사람이 자연어로 묻는 질문을 SQL 쿼리로 번역해 실행하고 결과를 다시 자연어로 설명합니다.

일반 SQL 에이전트와 다른 점은 쿼리를 실행 전에 LLM이 한 번 더 검수하고, 실행 중 오류가 나면 같은 노드로 되돌아가 쿼리를 고쳐 다시 시도한다는 것입니다. 즉, 그래프 자체에 "실수 → 인지 → 수정"의 루프가 내장돼 있습니다.

01에이전트가 따르는 7단계

튜토리얼 서두에서 제시하는 전체 흐름입니다. 이 7단계가 그대로 아래 그래프의 노드들로 구현됩니다.

  1. 데이터베이스 스키마 파악 — 사용 가능한 테이블 목록을 가져온다.
  2. 관련 테이블 선택 — 질문과 연관된 테이블만 고른다.
  3. DDL 조회 — 선택된 테이블의 스키마 정의(DDL)를 가져온다.
  4. 쿼리 생성 — 질문과 DDL을 근거로 SQL 쿼리를 작성한다.
  5. 쿼리 점검 — LLM으로 흔한 SQL 실수를 검토·수정한다.
  6. 쿼리 실행 및 오류 처리 — 실행하고, 오류가 나면 고쳐서 재시도한다.
  7. 응답 생성 — 쿼리 결과를 바탕으로 최종 답을 만든다.

02준비물 — DB와 4가지 도구

실습은 .env로 API 키를 불러오고 LangSmith 프로젝트(CH17-LangGraph-Use-Cases)로 트레이싱을 켠 뒤, GPT-4o(MODEL_NAME) 모델과 공개 URL에서 내려받은 Chinook.db SQLite 파일로 시작합니다.

SQLDatabaseToolkit이 제공하는 도구 2종과, 직접 정의한 도구·체인 2종을 조합해 사용합니다.

list_tables_tool

데이터베이스에 존재하는 모든 테이블 이름을 나열. 그래프의 첫 번째 강제 호출 대상.

sql_db_list_tables
get_schema_tool

지정한 테이블(들)의 DDL(CREATE TABLE 정의)을 반환. 컬럼명·타입을 정확히 알기 위해 필요.

sql_db_schema
db_query_tool

쿼리를 실제로 실행하는 커스텀 도구. 결과가 없거나 실패하면 예외 대신 "Error: ..." 문자열을 반환해 에이전트가 읽을 수 있게 한다.

@tool db_query_tool(query: str)
query_check

쿼리를 실행하기 직전, NULL 비교·JOIN 컬럼 오류 등 흔한 SQL 실수를 LLM이 검토·수정. 도구는 아니고 그래프 노드로 쓰인다.

query_check_prompt | llm.bind_tools([db_query_tool])

또한 create_tool_node_with_fallback()이 모든 ToolNode를 감싸, 도구 실행 중 예외가 나도 그래프가 멈추지 않고 오류 내용을 ToolMessage로 에이전트에 되돌려주도록 합니다.

03상태(State) — 메시지 하나로 충분

CRAG처럼 여러 키를 나누지 않고, 대화 메시지 리스트 하나에 도구 호출·결과·중간 판단을 전부 누적하는 방식입니다. 다음에 무엇을 할지는 마지막 메시지의 내용으로 판단합니다.

messages Annotated[list[AnyMessage], add_messages]
사용자 질문 → 도구 호출 → 도구 결과 → 생성된 쿼리 → 실행 결과 → 최종 답변까지, 모든 단계가 이 하나의 리스트에 순서대로 쌓인다. add_messages가 새 메시지를 자동으로 append.

04전체 그래프 흐름

위쪽 4단계(회색·파랑)는 스키마를 파악하는 준비 과정이고, 가운데 query_gen이 전체를 지휘하는 허브입니다. 마지막 메시지가 무엇이냐에 따라 세 갈래 중 하나로 흐릅니다.

START
first_tool_call
sql_db_list_tables 호출을 강제로 삽입
list_tables_tool
DB의 전체 테이블 목록 반환
model_get_schema
질문 + 테이블 목록 → 조회할 테이블 결정 후 get_schema_tool 호출
get_schema_tool
선택된 테이블의 DDL(스키마) 반환
query_gen · 분기 허브
마지막 메시지를 보고 새 쿼리 작성 / 오류 재작성 / 최종 답변 중 하나를 결정
correct_query
query_check로 쿼리 재검수
execute_query
db_query_tool로 실제 실행
END
그 외(SQL문) → 검수
실행 결과 → 재검토
Error: → 재생성
Answer: → 완료
스키마 탐색 단계 query_gen (분기 허브) 검수 · 실행 루프 오류 재생성 최종 답변

05노드 구성 한눈에 보기

각 노드가 정확히 무엇을 하는지 표로 정리했습니다.

노드하는 일
first_tool_call 실제 LLM 호출 없이, sql_db_list_tables를 호출하는 AIMessage를 직접 만들어 넣는다 (첫 단계 고정)
list_tables_tool ToolNode로 감싼 list_tables_tool 실행 → 테이블 목록 확보
model_get_schema get_schema_tool에 bind_tools한 LLM. 질문과 테이블 목록을 보고 어떤 테이블의 DDL이 필요한지 판단해 도구를 호출
get_schema_tool ToolNode로 감싼 get_schema_tool 실행 → 선택된 테이블의 DDL 확보
query_gen 지금까지의 메시지를 바탕으로 SQL 쿼리 생성, 오류 메시지 그대로 반환, 또는 최종 답변(Answer: ...) 중 하나를 출력
correct_query model_check_query 실행 — query_check 체인으로 쿼리를 한 번 더 검수하고 db_query_tool 호출을 강제
execute_query ToolNode로 감싼 db_query_tool 실행 → 쿼리 결과 또는 "Error: ..." 반환

06라우팅 로직 — query_gen이 갈림길을 정하는 법

프롬프트(QUERY_GEN_INSTRUCTION)는 query_gen에게 아래 4가지 규칙 중 하나로만 응답하라고 지시합니다. 그 응답의 내용 접두어를 should_continue가 읽어 다음 노드를 정합니다.

규칙 1 · 새 쿼리 작성

쓸 만한 실행 결과가 아직 없다면 SQLite 쿼리를 새로 만든다. DML(INSERT/UPDATE/DELETE/DROP 등)은 절대 금지.

규칙 2 · 쿼리만 출력

쿼리를 만들었다면 다른 말 없이 쿼리 문자열만 응답한다 — 도구 호출 형태가 아니라 일반 텍스트로.

규칙 3 · 오류 그대로 반환

직전 실행이 실패했다면 같은 오류 메시지를 그대로 응답한다. 예: "Error: Pets table doesn't exist".

규칙 4 · 최종 답변

쿼리 결과가 이미 있다면 이를 해석해 "Answer: ..." 형태로 응답한다.

참고 · SubmitFinalAnswer와 잘못된 도구 호출

query_gen은 실제로는 final_answer: str 필드 하나짜리 SubmitFinalAnswer 라는 pydantic 도구와 model_check_query를 bind_tools([SubmitFinalAnswer, model_check_query])로 함께 바인딩하고 있습니다. query_gen_node는 모델이 SubmitFinalAnswer가 아닌 다른 도구를 호출하면 "잘못된 도구가 호출됐다"는 ToolMessage 오류를 즉시 만들어 붙여 되돌립니다.

다만 실제 분기(should_continue)는 이 도구 호출 여부가 아니라, 위 규칙대로 만들어진 메시지 내용의 Answer:/Error: 접두어로 판단합니다 — 이 노트북에서 조금 헷갈리기 쉬운 지점입니다.

query_gen의 응답다음 노드의미
쿼리 문자열 그대로 correct_query 아직 실행할 만한 결과가 없음 → 새로 만든 SQL을 검수 단계로 보낸다
"Error: ..." query_gen (재시도) 직전 실행이 실패 → 같은 노드로 돌아가 고친 쿼리를 다시 생성
"Answer: ..." END 쿼리 결과 해석 완료 → 최종 답변으로 그래프 종료
should_continue() · query_gen 뒤의 조건부 라우팅
def should_continue(state: State) -> Literal[END, "correct_query", "query_gen"]:
    last_message = state["messages"][-1]

    if last_message.content.startswith("Answer:"):
        # 최종 답변 완료 → 종료
        return END
    if last_message.content.startswith("Error:"):
        # 직전 실행 실패 → query_gen으로 되돌아가 재작성
        return "query_gen"
    # 그 외(= 새 SQL 쿼리) → 실행 전 검수 단계로
    return "correct_query"

주의할 점은 execute_query → query_gen 엣지가 조건 없이 항상 연결되어 있다는 것입니다. 즉 실행이 성공하든 실패하든 일단 query_gen으로 돌아가고, query_gen이 그 결과(정상 데이터 또는 "Error: ...")를 보고 새로 만든 응답을 should_continue가 다시 읽어서 다음 길을 정합니다. "실행 → 바로 분기"가 아니라 "실행 → query_gen 재호출 → 그 결과로 분기"인 2단계 흐름인 셈이죠.

07그래프 조립 코드

준비 단계는 순차적 add_edge로, query_gen 이후만 add_conditional_edges로 갈라집니다.

workflow 구성 · StateGraph
workflow.add_edge(START, "first_tool_call")
workflow.add_edge("first_tool_call", "list_tables_tool")
workflow.add_edge("list_tables_tool", "model_get_schema")
workflow.add_edge("model_get_schema", "get_schema_tool")
workflow.add_edge("get_schema_tool", "query_gen")

# query_gen 이후에만 세 갈래로 분기 (END / correct_query / query_gen)
workflow.add_conditional_edges("query_gen", should_continue)

workflow.add_edge("correct_query", "execute_query")
workflow.add_edge("execute_query", "query_gen")  # 실행 후 다시 query_gen으로

app = workflow.compile(checkpointer=MemorySaver())

08실행 예시 & LangSmith 평가

run_graph()로 질문 두 가지를 직접 실행해보고, 이어서 LangSmith 데이터셋 + LLM-as-judge로 에이전트 응답의 정확도를 채점합니다.

run_graph()는 내부적으로 RunnableConfig(recursion_limit=30, configurable={"thread_id": random_uuid()})를 구성해 재귀 한도와 MemorySaver 체크포인트용 대화 스레드를 지정하고, 한도를 넘으면 GraphRecursionError를 잡아 그 시점까지의 상태를 반환합니다. 재시도 루프가 있는 그래프이기 때문에 필요한 안전장치입니다.

실행 예시 1 — 특정 레코드 조회

질문: "Andrew Adam 직원의 인적정보를 모두 조회해줘"

list_tables → get_schema(Employee) → query_gen → correct_query → execute_query → Answer

Employee 테이블 스키마만 확인하면 되는 단일 테이블 조회라, 재시도 없이 한 번에 정상 경로를 통과하기 쉬운 케이스입니다.

실행 예시 2 — 집계·조인이 필요한 질문

질문: "2009년도에 어느 국가의 고객이 가장 많이 지출했을까요? 그리고 얼마를 지출했을까요?"

list_tables → get_schema(Customer, Invoice ...) → query_gen → correct_query → execute_query → Answer

날짜 필터(2009)와 여러 테이블 조인 + 그룹 집계가 필요해 쿼리가 복잡해지고, 컬럼명을 잘못 짚으면 query_gen이 Error:를 받아 재시도 루프를 타기 쉬운 케이스입니다.

이후 LangSmith에 데이터셋 "SQL Agent Response"를 만들어, 노트북의 examples 리스트에 있는 아래 5개 질문·정답 쌍(원문 그대로 인용)으로 answer_evaluator(LLM-as-judge, langchain-ai/rag-answer-vs-reference 프롬프트 사용)가 에이전트 응답과 정답을 비교 채점합니다.

질문 (input)정답 (reference)
Which country's customers spent the most? And how much did they spend?The country whose customers spent the most is the USA, with a total spending of 523.06.
What was the most purchased track of 2013?The most purchased track of 2013 was Hot Girl.
How many albums does the artist Led Zeppelin have?Led Zeppelin has 14 albums
What is the total price for the album “Big Ones”?The total price for the album 'Big Ones' is 14.85
Which sales agent made the most in sales in 2009?Steve Johnson made the most sales in 2009

평가는 evaluate()로 num_repetitions=3회 반복 실행되어, 에이전트의 답변 재현성까지 함께 확인합니다.

09핵심 정리

  • 이 에이전트는 스키마 탐색(4단계) → 쿼리 생성·검수·실행 루프 두 부분으로 이루어진다.
  • 상태는 messages 리스트 하나뿐이며, 다음 행동은 마지막 메시지의 내용으로 결정된다.
  • query_gen은 응답을 Answer: / Error: / 그 외(쿼리)로 나눠 세 갈래 경로를 만드는 분기 허브다.
  • 실행 실패는 그래프를 멈추지 않고 query_gen으로 되돌아가는 재시도 루프로 흡수된다.
  • create_tool_node_with_fallback이 도구 예외를 ToolMessage로 바꿔주기 때문에 이 루프가 가능하다.
  • 완성된 에이전트는 LangSmith 데이터셋 + LLM-as-judge로 답변 정확도를 정량 평가한다.
