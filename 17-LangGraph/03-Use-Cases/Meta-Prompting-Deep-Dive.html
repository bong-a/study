Deep Dive · 실제 상수 해부

OpenAI가 공개한 META_PROMPT과 META_SCHEMA를 한 줄씩 뜯어보기

이 두 상수는 실제로 OpenAI Playground의 "Generate" 버튼 뒤에서 도는 텍스트입니다. 문서에 전문이 그대로 공개되어 있어 그대로 복사해 쓸 수 있습니다. 여기서는 각 규칙이 무슨 뜻이고 왜 있는지를 붙여 읽습니다.

PART A — 텍스트 프롬프트 생성기

META_PROMPT 전문 & 해부

먼저 원문 그대로입니다. 우측 copy 버튼으로 복사해 바로 system 메시지에 넣어 쓸 수 있습니다.

META_PROMPT (verbatim) 
Given a task description or existing prompt, produce a detailed system prompt to guide a language model in completing the task effectively.

# Guidelines

- Understand the Task: Grasp the main objective, goals, requirements, constraints, and expected output.
- Minimal Changes: If an existing prompt is provided, improve it only if it's simple. For complex prompts, enhance clarity and add missing elements without altering the original structure.
- Reasoning Before Conclusions**: Encourage reasoning steps before any conclusions are reached. ATTENTION! If the user provides examples where the reasoning happens afterward, REVERSE the order! NEVER START EXAMPLES WITH CONCLUSIONS!
    - Reasoning Order: Call out reasoning portions of the prompt and conclusion parts (specific fields by name). For each, determine the ORDER in which this is done, and whether it needs to be reversed.
    - Conclusion, classifications, or results should ALWAYS appear last.
- Examples: Include high-quality examples if helpful, using placeholders [in brackets] for complex elements.
   - What kinds of examples may need to be included, how many, and whether they are complex enough to benefit from placeholders.
- Clarity and Conciseness: Use clear, specific language. Avoid unnecessary instructions or bland statements.
- Formatting: Use markdown features for readability. DO NOT USE ``` CODE BLOCKS UNLESS SPECIFICALLY REQUESTED.
- Preserve User Content: If the input task or prompt includes extensive guidelines or examples, preserve them entirely, or as closely as possible. If they are vague, consider breaking down into sub-steps. Keep any details, guidelines, examples, variables, or placeholders provided by the user.
- Constants: DO include constants in the prompt, as they are not susceptible to prompt injection. Such as guides, rubrics, and examples.
- Output Format: Explicitly the most appropriate output format, in detail. This should include length and syntax (e.g. short sentence, paragraph, JSON, etc.)
    - For tasks outputting well-defined or structured data (classification, JSON, etc.) bias toward outputting a JSON.
    - JSON should never be wrapped in code blocks (```) unless explicitly requested.

The final prompt you output should adhere to the following structure below. Do not include any additional commentary, only output the completed system prompt. SPECIFICALLY, do not include any additional messages at the start or end of the prompt. (e.g. no "---")

[Concise instruction describing the task - this should be the first line in the prompt, no section header]

[Additional details as needed.]

[Optional sections with headings or bullet points for detailed steps.]

# Steps [optional]

[optional: a detailed breakdown of the steps necessary to accomplish the task]

# Output Format

[Specifically call out how the output should be formatted, be it response length, structure e.g. JSON, markdown, etc]

# Examples [optional]

[Optional: 1-3 well-defined examples with placeholders if necessary. Clearly mark where examples start and end, and what the input and output are. User placeholders as necessary.]
[If the examples are shorter than what a realistic example is expected to be, make a reference with () explaining how real examples should be longer / shorter / different. AND USE PLACEHOLDERS! ]

# Notes [optional]

[optional: edge cases, details, and an area to call or repeat out specific important considerations]

# Guidelines 9개 지침 해부

이 프롬프트의 심장부입니다. 각 지침을 무슨 뜻 / 왜로 풀면:

지침 (원문)무슨 뜻 · 왜
01Understand the Task주 목표·요구사항·제약·기대 출력을 먼저 파악. → 프롬프트를 짜기 전에 "무엇을 시키는가"부터 정의.
02Minimal Changes기존 프롬프트가 있으면 단순할 때만 개선. 복잡하면 원래 구조를 건드리지 말고 명확성·누락 요소만 보강. → 잘 돌던 걸 망치지 않기.
03Reasoning Before Conclusions결론 전에 추론. 사용자 예시가 "결론 먼저" 구조면 순서를 뒤집으라고까지 강제(REVERSE!). 결론·분류·결과는 항상 마지막. → 메타 프롬프트 전체 철학의 핵심.
04Examples도움되면 고품질 예시 포함, 복잡한 부분은 [대괄호] 플레이스홀더로. 몇 개·얼마나 복잡한지 판단.
05Clarity & Conciseness명확·구체적 언어. 불필요한 지시나 밋밋한 문장 금지.
06Formatting가독성 위해 마크다운 사용. 단, 명시 요청 없으면 ``` 코드블록 쓰지 말 것.
07Preserve User Content사용자가 준 방대한 가이드라인·예시는 최대한 그대로 보존. 모호하면 하위 단계로 분해. 변수·플레이스홀더도 유지.
08Constants가이드·루브릭·예시 같은 상수는 포함하라 — 상수는 프롬프트 인젝션에 안전하기 때문.
09Output Format가장 적절한 출력 형식을 상세히 명시(길이·구문). 구조화 데이터는 JSON 편향. JSON은 명시 요청 없으면 코드블록으로 감싸지 말 것.
TIP

03번의 "REVERSE the order!"가 백미입니다. 사용자가 무심코 "정답 → 이유" 순서의 예시를 넣어도, 메타 프롬프트가 이를 감지해 "이유 → 정답" 순서로 강제 교정합니다. 사용자의 실수까지 방어하는 설계.

structure 출력 구조 템플릿 해부

# Guidelines 아래는 "이 형식으로 뱉어라"는 출력 골격입니다.

블록역할
첫 줄 (헤더 없음)작업을 설명하는 간결한 지시문. 반드시 첫 줄, 섹션 제목 없이.
# Steps (옵션)작업 수행에 필요한 단계별 분해.
# Output Format출력 형식을 콕 집어 명시(길이·구조·JSON/마크다운 등).
# Examples (옵션)1~3개 예시 + 플레이스홀더. 실제보다 짧으면 ()로 "실전은 더 길다"고 표기.
# Notes (옵션)엣지 케이스·중요 고려사항 반복 강조.
META

재귀의 묘미: META_PROMPT 자신이 자기 규칙을 그대로 지킵니다 — 지침(추론)을 먼저 두고 출력 구조(결론)를 마지막에 배치하고, 상수(가이드라인)를 본문에 박아뒀죠. "프롬프트 잘 쓰는 법"을 스스로 시연하는 프롬프트입니다.

주의

원문에는 Reasoning Before Conclusions**처럼 다듬어지지 않은 마크다운 흔적(**)이나 User placeholders(→ Use 오타로 보임) 같은 자국이 그대로 있습니다. 완벽한 문서가 아니라 실사용 상수를 그대로 노출한 것이므로, 복사해 쓸 때 참고하세요.

PART B — 구조화 출력 스키마 생성기

META_SCHEMA 전문 & 해부

이건 프롬프트가 아니라 JSON 스키마입니다. 정확히는 "스키마가 어떻게 생겨야 하는지를 설명하는 스키마" — 그래서 meta-schema죠.

META_SCHEMA (verbatim) 
{
  "name": "metaschema",
  "schema": {
    "type": "object",
    "properties": {
      "name": { "type": "string", "description": "The name of the schema" },
      "type": { "type": "string", "enum": ["object", "array", "string", "number", "boolean", "null"] },
      "properties": {
        "type": "object",
        "additionalProperties": { "$ref": "#/$defs/schema_definition" }
      },
      "items": {
        "anyOf": [
          { "$ref": "#/$defs/schema_definition" },
          { "type": "array", "items": { "$ref": "#/$defs/schema_definition" } }
        ]
      },
      "required": { "type": "array", "items": { "type": "string" } },
      "additionalProperties": { "type": "boolean" }
    },
    "required": ["type"],
    "additionalProperties": false,
    "if": { "properties": { "type": { "const": "object" } } },
    "then": { "required": ["properties"] },
    "$defs": {
      "schema_definition": {
        // ↑ 위 schema와 동일한 형태를 그대로 반복 — 이것이 "재귀 엔진"
        "type": "object",
        "properties": { "type", "properties", "items", "required", "additionalProperties" },
        "required": ["type"],
        "additionalProperties": false,
        "if": { "properties": { "type": { "const": "object" } } },
        "then": { "required": ["properties"] }
      }
    }
  }
}

$defs.schema_definition 내부는 최상위 schema와 완전히 동일한 구조라 위에서 필드명만 축약했습니다. 전체 원문이 필요하면 copy 버튼으로 복사되는 텍스트는 두 블록 모두 완전한 형태입니다.

structure 한눈에 보는 구조

metaschema object · additionalProperties: false (닫힘)
필드타입설명
namestring스키마 이름
type 필수enum(6)object · array · string · number · boolean · null
propertiesmap각 필드의 값 = $defs/schema_definition (재귀)
itemsanyOf단일 요소 또는 튜플(배열)의 요소 타입
requiredstring[]필수 필드 목록
additionalPropertiesboolean정의 밖 필드 허용 여부
규칙 1
required: ["type"]모든 노드는 type을 반드시 선언
규칙 2
if type == "object" → then "properties"객체면 properties 필수
재귀 엔진
$defs/schema_definition자기 자신과 같은 형태 → 무한 중첩 표현 가능
핵심

$defs/schema_definition이 재귀의 열쇠입니다. properties의 각 값도, items의 요소도 모두 자기 자신(schema_definition)을 $ref로 가리키죠. 덕분에 객체 안의 객체 안의 배열… 같은 임의 깊이의 중첩 스키마를 표현할 수 있습니다. if/then은 "object라고 했으면 properties를 꼭 정의하라"는 강제 규칙.

why 왜 이게 "유사(pseudo)" 메타 스키마인가

개념 가이드에서 나온 "닭이 먼저냐 달걀이 먼저냐" 딜레마가 여기서 구체화됩니다. 이 META_SCHEMA는 strict 모드 스키마를 만들어내지만, 정작 자기 자신은 strict 모드가 금지한 기능들을 씁니다:

META_SCHEMA가 쓰는 기능왜 strict 모드 위반인가
additionalProperties: { $ref }고정 키가 아닌 임의 키의 맵(dictionary)을 표현 — strict 모드는 additionalProperties: false만 허용.
anyOf (items)여러 형태 중 택일 — 최상위 스키마 조합에 제약.
if / then / const조건부 검증 키워드 — strict 모드 미지원.
딜레마

즉 "스키마를 만드는 도구(meta-schema)"는 non-strict로 굴리고, 그 결과물인 스키마는 생성 후 검증 단계에서 strict 모드에 맞게 정리(cleaning)합니다. 그래서 이름이 pseudo-meta-schema — "만들 땐 느슨, 결과는 엄격".

strict 생성된 스키마가 피해야 할 키워드

스키마 생성용 메타 프롬프트는 결과 스키마에서 아래 미지원 키워드를 쓰지 않도록 명시합니다.

unevaluatedProperties propertyNames minProperties maxProperties unevaluatedItems contains minContains

그리고 결과 스키마는 반드시: 모든 필드 required · 모든 객체 additionalProperties: false · 모든 객체는 properties 정의 · 추론 필드가 결론 필드보다 먼저. (META_PROMPT의 03번 원칙이 스키마 세계에서도 반복됩니다.)

예시

스키마 생성 메타 프롬프트에는 3개의 데모 예시가 들어 있습니다 — ① 수학 추론 스키마(단계별 풀이 → 최종답), ② 연결 리스트(linked list)의 재귀 표현, ③ 동적 생성 UI 컴포넌트 스키마. 재귀·순서·동적 구조를 모두 보여주는 셈.

PART C — 종합

둘은 이렇게 맞물린다

사용자 입력 "설명"
텍스트 필요
▸system = META_PROMPT
→완성된 프롬프트(문자열)
구조화 출력 필요
▸system = META_PROMPT
▸response_format = META_SCHEMA
→JSON 스키마 → strict 정리
  • META_PROMPT = "좋은 프롬프트/스키마를 만드는 방법론"을 담은 지시문 (system 역할).
  • META_SCHEMA = 스키마 생성 시 결과물의 형태를 강제하는 틀 (response_format 역할).
  • 둘 다 관통하는 원칙 하나 — "추론 먼저, 결론은 항상 마지막".
