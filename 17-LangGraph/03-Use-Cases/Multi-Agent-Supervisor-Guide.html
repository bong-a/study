세 줄 정리

① 문제

에이전트가 2~3개를 넘어가면, 이전 결과에 따라 다음 노드를 하드코딩된 조건문으로 분기하는 방식은 규칙이 기하급수적으로 늘어나 관리가 어려워진다.

② 해법

Supervisor라는 별도의 LLM 노드가 대화 흐름 전체를 보고, 구조화된 출력(Literal)으로 다음에 일할 작업자를 매번 새로 판단한다.

③ 효과

새 전문 에이전트를 추가해도 라우팅 조건문을 다시 짤 필요 없이 워커 노드를 등록하고 members 목록과 Supervisor 프롬프트만 갱신하면 된다 — 확장 가능한 패턴.

01 · 배경

왜 단순 분기 로직으로는 부족한가

이전 튜토리얼(에이전트 협업, Agent Collaboration)에서는 Researcher 에이전트의 출력을 보고 조건부 엣지로 다음 노드를 결정했다. 에이전트가 하나 더 늘어날 때마다 분기 조건도 함께 늘어나는 구조다.

문제 지점: "누가 다음에 일할지"를 결정하는 로직이 코드에 하드코딩되어 있어, 팀 구성이 바뀔 때마다 그래프 로직을 뜯어고쳐야 한다.

Supervisor 아이디어

다음 작업자를 결정하는 책임을 별도의 LLM 노드(Supervisor)에게 위임한다. Supervisor는 지금까지의 대화(messages)를 보고 "누가 다음에 일할지, 혹은 끝낼지"를 구조화된 출력으로 답한다. 새 에이전트가 늘어나도 새 워커 노드를 등록한 뒤 Supervisor의 멤버 목록을 갱신하면 라우팅 구조는 그대로 유지된다.

02 · 개요

Supervisor 아키텍처 한눈에

Supervisor는 팀의 허브(hub) 역할을 한다. 모든 작업자(Researcher, Coder)는 작업이 끝나면 결과를 Supervisor에게 보고하고, Supervisor는 그 결과를 보고 다음 작업자를 다시 지명하거나 작업을 종료(FINISH)한다.

실선(teal) = Supervisor가 작업을 할당하는 조건부 엣지 · 점선(indigo) = 작업자가 결과를 보고하는 일반 엣지 · 빨간선 = 종료 엣지.

03 · 상태 정의

AgentState — 팀이 공유하는 하나의 상태

멀티 에이전트 시스템에서는 모든 노드가 동일한 상태 객체를 읽고 쓴다. 이 튜토리얼의 상태는 단 두 개의 필드로 이루어진다.

필드타입설명
messagesAnnotated[Sequence[BaseMessage], operator.add]지금까지 오간 모든 메시지. operator.add로 새 메시지가 누적된다.
nextstrSupervisor가 결정한 다음에 실행할 노드 이름 (또는 FINISH).
 
class AgentState(TypedDict):
    messages: Annotated[Sequence[BaseMessage], operator.add]  # 메시지 누적
    next: str  # 다음으로 라우팅할 에이전트
왜 operator.add 인가

각 노드는 상태 전체를 새로 만드는 대신 변경분만 반환한다. LangGraph는 반환된 messages 리스트를 기존 상태의 messages에 operator.add(리스트 덧셈)로 이어 붙인다. 그 결과 Researcher와 Coder의 발언이 대화 기록에 그대로 쌓이고, Supervisor는 매 턴 전체 히스토리를 보고 판단할 수 있다.

04.5 · 헬퍼 유틸리티

agent_node + functools.partial — 노드를 간결하게

모든 워커 노드는 같은 패턴을 반복한다 — agent를 호출하고, 마지막 메시지를 이름표를 붙여 상태에 반환. 이 반복을 agent_node 함수 하나로 추출한다.

def agent_node(state, agent, name):
    agent_response = agent.invoke(state)
    return {
        "messages": [
            HumanMessage(content=agent_response["messages"][-1].content, name=name)
        ]
    }

functools.partial로 노드 만들기

매번 agent와 name을 넘기는 대신, functools.partial로 값을 미리 고정한 전용 함수를 만든다.

방식호출 코드비고
Beforeagent_node(state, agent=research_agent, name="Researcher")호출할 때마다 agent·name을 매번 지정
Afterresearch_node(state)functools.partial로 agent·name이 이미 고정된 노드 함수
 
research_node = functools.partial(agent_node, agent=research_agent, name="Researcher")
coder_node = functools.partial(agent_node, agent=coder_agent, name="Coder")

덕분에 workflow.add_node("Researcher", research_node)처럼 그래프에 바로 등록할 수 있는 깔끔한 노드 함수가 만들어진다.

※ 원본 노트북의 functools.partial 설명용 마크다운에는 names="Researcher"라고 적힌 예시가 있지만, 실제 실행 코드와 agent_node 함수 시그니처는 name="Researcher"가 맞다.

05 · 감독자 생성

Supervisor Agent — 구조화된 출력으로 다음 행동 결정

Supervisor는 일반 텍스트가 아니라 정해진 선택지 중 하나만 반환해야 한다. 이를 위해 Pydantic 모델과 with_structured_output을 사용한다.

2 + 1

Literal
RouteResponse.next 타입

Supervisor LLM 설정
라우팅 결정의 일관성 확보
members = ["Researcher", "Coder"]
options_for_next = ["FINISH"] + members

class RouteResponse(BaseModel):
    next: Literal[*options_for_next]  # "FINISH" | "Researcher" | "Coder"

Supervisor 프롬프트 구성

구성 요소내용
system (역할 지정)"당신은 다음 워커들 간의 대화를 관리하는 감독자다: {members}. 다음에 행동할 워커로 응답하라. 끝나면 FINISH로 응답하라."
MessagesPlaceholder지금까지 누적된 messages 전체를 프롬프트에 삽입
system (마무리 질문)"위 대화를 보고, 다음은 누가 행동해야 하는가? 혹은 FINISH해야 하는가? {options} 중 하나를 선택하라."
 
prompt = ChatPromptTemplate.from_messages([
    ("system", system_prompt),
    MessagesPlaceholder(variable_name="messages"),
    ("system", "Given the conversation above, who should act next? "
               "Or should we FINISH? Select one of: {options}"),
]).partial(options=str(options_for_next), members=", ".join(members))

llm = ChatOpenAI(model=MODEL_NAME, temperature=0)

partial(...)은 선택 가능한 워커 목록과 FINISH 옵션을 프롬프트에 고정한다. 그래서 Supervisor는 매 턴 자유 문장이 아니라 허용된 라우팅 후보 중 하나를 고르게 된다.

→

| "Coder" | "FINISH"
def supervisor_agent(state):
    supervisor_chain = prompt | llm.with_structured_output(RouteResponse)
    return supervisor_chain.invoke(state)

반환값이 그대로 AgentState의 next 필드로 들어가, 조건부 엣지가 이 값을 보고 다음 노드를 정한다.

06 · 그래프 구성

노드와 엣지로 그래프 조립하기

지금까지 만든 research_node, coder_node, supervisor_agent를 StateGraph에 등록하고 엣지로 연결한다.

엣지 종류연결결정 방식
시작 엣지START → Supervisor고정
일반 엣지Researcher → Supervisor
Coder → Supervisor		고정 (작업 후 항상 보고)
조건부 엣지Supervisor → {Researcher | Coder | END}state["next"] 값으로 분기
 
workflow = StateGraph(AgentState)
workflow.add_node("Researcher", research_node)
workflow.add_node("Coder", coder_node)
workflow.add_node("Supervisor", supervisor_agent)

for member in members:
    workflow.add_edge(member, "Supervisor")

conditional_map = {k: k for k in members}
conditional_map["FINISH"] = END

def get_next(state):
    return state["next"]

workflow.add_conditional_edges("Supervisor", get_next, conditional_map)
workflow.add_edge(START, "Supervisor")

graph = workflow.compile(checkpointer=MemorySaver())
conditional_map 이해하기

{"Researcher": "Researcher", "Coder": "Coder", "FINISH": END} — Supervisor가 반환한 문자열을 그대로 다음 노드 이름으로 매핑한다. 원본 코드는 members로 이 딕셔너리를 만들기 때문에, 워커가 늘어나면 새 노드를 등록하고 members에 이름을 추가하면 된다.

from langchain_teddynote.graphs import visualize_graph

visualize_graph(graph)

원본 노트북은 컴파일된 그래프를 바로 시각화한다. 위쪽의 원본 다이어그램 이미지는 이 단계에서 확인하는 그래프 구조를 학습용으로 다시 배치한 것이다.

※ MemorySaver는 메모리 기반 체크포인터다. 같은 프로세스와 같은 체크포인터 안에서 동일한 thread_id를 쓰면 이전 체크포인트를 이어갈 수 있고, 원본처럼 random_uuid()를 쓰면 실행마다 새 대화 스레드로 시작한다.

07 · 팀 호출

실행 흐름 — GDP 그래프 요청의 대표 경로

입력: "2010년 ~ 2024년까지의 대한민국의 1인당 GDP 추이를 그래프로 시각화 해주세요." 원본 노트북은 실행 결과 로그를 저장하지 않으므로, 아래는 이 요청에서 기대되는 전형적인 분배 흐름이다. 실제 라우팅은 모델 응답과 검색 결과에 따라 달라질 수 있다.

1
START → Supervisor

요청 접수 및 라우팅 판단

사용자 메시지를 확인한 Supervisor가 "데이터부터 조사해야 한다"고 판단하고 next = "Researcher"를 반환한다.

2
Researcher 실행

웹 검색으로 GDP 데이터 수집

TavilySearch로 연도별 1인당 GDP 자료를 검색하고, 결과를 HumanMessage(name="Researcher")로 감싸 상태에 추가한 뒤 Supervisor에게 돌아간다.

3
Supervisor 재판단

다음 단계는 시각화

대화 기록에 쌓인 검색 결과를 확인한 Supervisor가 next = "Coder"를 반환한다.

4
Coder 실행

matplotlib 코드 작성 및 실행

PythonREPLTool로 그래프 코드를 작성·실행한다. code_system_prompt는 생성 코드에 OS별 한글 폰트(맑은 고딕 / AppleGothic / NanumGothic) 설정을 포함하도록 지시한다.

5
Supervisor → FINISH

작업 완료 판단

그래프까지 생성되었음을 확인한 Supervisor가 next = "FINISH"를 반환 → conditional_map에 의해 END로 종료된다.

config = RunnableConfig(recursion_limit=10, configurable={"thread_id": random_uuid()})

inputs = {
    "messages": [
        HumanMessage(content="2010년 ~ 2024년까지의 대한민국의 1인당 GDP 추이를 그래프로 시각화 해주세요.")
    ],
}

invoke_graph(graph, inputs, config)

recursion_limit=10 — Supervisor ↔ 워커를 오가는 루프가 무한히 반복되지 않도록 최대 스텝 수를 제한한다.

08 · 비교 분석

단순 조건부 라우팅 vs Supervisor 패턴

이전 튜토리얼의 "에이전트 출력 기반 분기"와 이번 Supervisor 패턴을 나란히 놓고 비교하면 확장성의 차이가 분명해진다.

항목 단순 조건부 라우팅 Supervisor 패턴
라우팅 결정 주체 ✕ 코드에 하드코딩된 if/else ✓ 별도 LLM 노드(Supervisor)
신규 에이전트 추가 ✕ 분기 로직 재작성 필요 ✓ 새 워커 노드 등록 + members 목록 갱신
확장성 ~ 워커 2~3개까지는 무난, 이후 급격히 복잡 ✓ 워커 수가 늘어도 구조 동일
라우팅 근거 표현력 ~ 정해진 필드/키워드 매칭 수준 ✓ 전체 대화 맥락을 이해하고 판단
추가 비용 ✓ 없음 ✕ 매 스텝마다 Supervisor LLM 호출 필요
동작 예측 가능성 ✓ 결정적(deterministic) ~ temperature=0이어도 LLM 라우팅은 실행 환경과 응답에 따라 달라질 수 있음

한 단계 더 — 팀의 팀 (Hierarchical Teams)

Supervisor 패턴은 재귀적으로 중첩할 수 있다. 여러 Supervisor 팀을 다시 상위 Supervisor가 관리하면, 대규모 조직처럼 계층적인 멀티 에이전트 시스템을 구성할 수 있다.

정리 · 언제 쓰나 · 유의점

✓ 이럴 때 쓰면 좋다

  • 협업할 전문 에이전트가 3개 이상이거나 계속 늘어날 예정일 때
  • 작업 순서가 고정되어 있지 않고 상황에 따라 달라져야 할 때
  • 각 에이전트의 결과를 사람처럼 판단해 다음 행동을 정해야 할 때
  • 새 에이전트를 추가할 때 라우팅 조건문 재작성을 피하고 싶을 때

⚠ 한계 · 유의점

  • Supervisor 호출 자체가 추가 LLM 비용·지연을 발생시킨다
  • 라우팅이 확률적이므로 잘못된 워커를 지명할 가능성이 있다
  • 무한 루프 방지를 위해 recursion_limit를 설정하는 것이 안전하다
  • 워커가 매우 많아지면 Supervisor 프롬프트의 멤버 설명이 비대해질 수 있다 (이때 계층적 팀 구조 고려)
한 줄 정리

단순 분기는 "정해진 길"만 갈 수 있지만, Supervisor는 매번 "지금 상황에 맞는 다음 걸음"을 LLM에게 묻는다. 팀이 커질수록 이 차이가 유지보수 비용의 격차로 드러난다.
