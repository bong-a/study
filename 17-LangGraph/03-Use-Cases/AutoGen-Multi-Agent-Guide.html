AutoGen

Enabling Next-Gen LLM Applications via Multi-Agent Conversation
대화하는 에이전트들로 복잡한 작업을 푸는 프레임워크

“하나의 거대한 에이전트 대신, 대화하는 여러 개의 작은 에이전트를.” 역할이 다른 대화형 에이전트(Conversable Agent)를 자유롭게 조합하고, 그 대화 흐름 자체를 자연어와 코드로 프로그래밍해 다양한 작업을 해결한다.

세 줄 정리

① 문제

단일 에이전트(AutoGPT)는 혼자 다 해야 하고, LangChain 같은 프레임워크는 단일-에이전트 파이프라인 중심이라 서로 다른 역할이 유연하게 협업하는 구조를 만들기 어렵다.

② 해법

역할이 다른 대화형 에이전트(LLM+Human+Tool 조합)를 정의하고, 에이전트 간 “대화”자체를 프로그래밍 대상으로 삼아 정적·동적 워크플로우를 모두 표현한다.

③ 효과

수학 문제풀이·RAG·의사결정·코딩·그룹챗·체스 등 6개 응용에서 기존 대비 뚜렷한 개선을 확인했다(예: MATH 69.48% vs GPT-4 단독 55.18%).

01 · 동기

왜 “대화”를 프로그래밍해야 하는가

2023년 당시 LLM 애플리케이션은 크게 두 갈래였다. 하나는 혼자서 계획·실행·평가를 다 떠맡는 단일 에이전트, 다른 하나는 미리 정해진 순서를 따르는 정적 파이프라인이다. 둘 다 “역할이 다른 여러 주체가 상황에 따라 유연하게 협업”하는 패턴을 표현하기 어렵다.

단일 에이전트 — AutoGPT류

계획, 도구 사용, 자기 평가를 한 에이전트가 전부 수행한다. 복잡한 작업일수록 방향을 잃기 쉽고, 역할을 나눠 서로 검증하게 만들 방법이 없다.

정적 파이프라인 — LangChain·CAMEL·MetaGPT류

LangChain의 에이전트는 기본적으로 단일-에이전트 패러다임이고, CAMEL·MetaGPT의 멀티에이전트는 역할과 대화 순서가 고정돼 있어 상황에 맞춰 협업 방식을 바꾸기 어렵다.

AutoGen의 관점

정답은 “더 똑똑한 한 에이전트”가 아니라, 서로 다른 역할의 에이전트들이 나누는 대화 그 자체를 프로그래밍 가능한 매체로 다루는 것이다. 누가 언제 무엇을 말할지를 자연어 프롬프트 + 코드로 유연하게 정의할 수 있어야 한다.

02 · 프레임워크

두 가지 핵심 설계 원칙

AutoGen은 Conversable Agent를 “특정 역할을 가지고, 다른 대화형 에이전트와 메시지를 주고받으며 내부 컨텍스트를 유지하는 개체”로 정의한다. 이 위에 두 가지 설계 원칙이 올라간다. 비유하자면, 서로 다른 직무를 맡은 팀원들이 하나의 채팅방에서 협업하는 모습과 비슷하다 — 다만 팀원 중 일부는 사람이고 일부는 LLM이며, 코드를 실행할 수 있는 팀원도 있다.

① Customizable Conversable Agents

하나의 에이전트는 아래 세 가지 능력을 자유롭게 조합해서 만들어진다. 역할마다 어떤 능력을 얼마나 섞을지가 곧 에이전트 설계다.

LLM-based

역할극(role-play), 상태 추론, 피드백을 받아 진전시키기, 코드 작성

Human-based

사람의 개입 빈도와 조건을 설정 가능한(configurable) human input

Tool-based

코드 실행, 함수 호출로 외부 환경과 상호작용

논문 정의

“An entity with a specific role that can pass messages to send and receive information to and from other conversable agents.”

② Conversation Programming

대화 흐름을 두 요소로 나눠 프로그래밍한다. Computation(무엇을 말할지)과 Control Flow(다음에 누가 말할지)다. 이 둘을 자연어와 파이썬 코드로 섞어 짤 수 있다는 게 핵심이다.

논문 인용

Computation — “conversation-centric actions that result in message passing.”

Control Flow — “agent decisions on which agents to send messages to are functions of inter-agent conversation.”

실제로는 이렇게 대화한다 — 예시

말로만 들으면 추상적이니, 짧은 대화 하나로 두 원칙을 동시에 보자. 사람이 계산을 요청하면 Assistant가 코드를 쓰고(Computation), 그 코드를 실행할 차례가 되면 자동으로 UserProxy에게 발화권이 넘어간다(Control Flow).

Human
"10에서 3을 뺀 값을 제곱해서 알려줘."
Assistant(LLM)
코드로 계산할게요.
result = (10 - 3) ** 2 print(result)
UserProxy(코드 실행)
실행 결과 → 49를 다시 대화에 회신
Assistant(LLM)
"계산 결과, 정답은 49입니다."

※ 이해를 돕기 위해 구성한 예시(논문에 실린 코드 원문은 아님). 4턴 모두 "메시지 전달"(Computation)이고, 각 턴이 끝날 때마다 "다음은 누구 차례인가"(Control Flow)가 자동으로 결정된다.

이 두 원칙 덕분에 같은 프레임워크로 “정해진 순서의 파이프라인”도, “상황에 따라 발화자가 바뀌는 그룹챗”도 만들 수 있다.

03 · 에이전트 구성

내장 에이전트 종류와 Human-in-the-loop

AutoGen은 바로 쓸 수 있는 몇 가지 내장 에이전트를 제공한다. 이들을 조합하면 별도 학습 없이도 다양한 워크플로우를 만들 수 있다.

에이전트역할특징
ConversableAgent최상위 추상화LLM·Human·Tool을 자유롭게 혼합 가능한 기반 클래스
AssistantAgentAI 어시스턴트LLM 기반으로 문제 해결·코드 작성을 담당
UserProxyAgent사람 대리인human input을 요청하거나, 대신 코드를 실행하고 결과를 회신
GroupChatManager그룹 대화 조율자다음 발화자를 동적으로 선택해 그룹챗을 진행

human_input_mode — 세 가지 자율성 수준

UserProxyAgent가 사람에게 얼마나 자주 확인을 받을지는 하나의 설정값으로 조절한다.

ALWAYS

매 라운드 확인

매 턴마다 human input을 요청한다. 가장 안전하지만 자동화 정도는 낮다.

TERMINATE

종료 조건에서만 확인

대화가 끝나려 할 때만 사람에게 확인을 받는다. 자동화와 안전의 중간 지점.

NEVER

완전 자동화

사람 개입 없이 끝까지 자동으로 진행한다. 자유롭지만 위험도 가장 크다.

논문의 가이드라인 (Appendix B.1)

“start with human_input_mode='ALWAYS', even if target is autonomous.” — 최종 목표가 완전 자동화라도, 우선 사람이 매번 확인하는 모드로 시작해 동작을 검증한 뒤 점진적으로 자동화 수준을 높이라고 권고한다.

04 · 코드 실행

Code Executor — 코드를 매개로 한 대화

AutoGen에서 “도구 사용”의 가장 대표적인 형태는 코드 실행이다. AssistantAgent가 코드를 작성하면, UserProxyAgent가 이를 실행하고 결과를 대화에 그대로 되돌려준다. 실패하면 그 결과가 다음 프롬프트가 되어 자연스럽게 수정 루프가 만들어진다.

에러가 나면 실제로 이런 대화가 오간다 — 예시

핵심은 실행 에러 메시지 자체가 다음 프롬프트가 된다는 점이다. 별도의 디버깅 로직 없이, "대화"의 다음 턴에 에러가 그대로 들어가는 것만으로 수정이 일어난다.

Human
"리스트 [1, 2, 0, 4]의 평균을 구해줘."
Assistant(LLM)
nums = [1, 2, 0, 4] avg = sum(nums) / len(numbers)
UserProxy(코드 실행)
NameError: name 'numbers' is not defined → 에러 메시지를 그대로 대화에 회신
Assistant(LLM)
에러를 보고 오타를 수정.
nums = [1, 2, 0, 4] avg = sum(nums) / len(nums)
UserProxy(코드 실행)
실행 결과 → 1.75

※ 이해를 돕기 위해 구성한 예시. 실제 논문의 A2 사례(FLAML+Spark)도 같은 원리 — 존재하지 않는 spark=True 파라미터를 생성했다가, 검색된 문서를 참고해 use_spark=True로 스스로 고친다.

Enhanced LLM inference layer

  • result caching — 동일 요청 재호출 비용 절감
  • error handling — API 오류·rate limit에 대한 재시도 처리
  • message templating — 프롬프트 조립을 표준화

※ 코드 실행은 제한된(sandboxed) 환경에서 이뤄지며, 안전성이 중요한 워크플로우에서는 이후 소개할 Safeguard 에이전트처럼 별도 검증 단계를 추가하는 패턴이 쓰인다.

05 · 응용 사례

6가지 응용으로 검증하다

논문은 서로 다른 도전 과제를 지닌 6개 응용에서 AutoGen을 실증한다. 같은 프레임워크로 “정적 2-에이전트 협업”부터 “동적 그룹챗”까지 모두 구현할 수 있음을 보여주는 것이 목적이다.

A1

수학 문제 풀이

AssistantAgent + UserProxyAgent

MATH 데이터셋에서 내장 에이전트만으로 GPT-4 단독, LangChain ReAct, Multi-Agent Debate 등을 모두 능가.

A2

검색증강 생성·QA

Retrieval-augmented UserProxy + Assistant

컨텍스트가 부족하면 스스로 “UPDATE CONTEXT”를 요청하는 Interactive Retrieval 도입.

A3

텍스트 세계 의사결정

Assistant + Executor (+ Grounding)

ALFWorld 134개 미지 과제. 예: "사과를 확인하라"는 지시에 Executor가 물건을 집지도 않고 바로 살펴보려다 실패 → Grounding 에이전트가 “You must find and take the object before examining it”라고 알려줘 성공. 이 3-Agent 설계가 평균 약 15% 성능 향상을 냄.

A4

멀티에이전트 코딩 (OptiGuide)

Commander + Writer + Safeguard

최적화 솔루션 해석 시스템. 코드량을 430줄→100줄로 줄이면서 안전성 검증까지 분리.

A5

동적 그룹챗

GroupChatManager + N개 에이전트

여러 전문가가 모인 단체 대화방을 떠올리면 쉽다 — 정해진 발언 순서 없이, 사회자(GroupChatManager)가 매 순간 대화 맥락을 보고 "지금 누가 답할 차례인지"를 정한다. Role-play 프롬프트가 순수 과제형 프롬프트보다 성공률이 높았다.

A6

대화형 체스

Player agents + Board agent

예: Player가 규칙에 어긋나는 수를 제안 → Board agent가 즉시 반려 → Player가 합법적인 수로 다시 제안(grounding). Board agent를 빼면 반칙으로 게임이 중단되는 사례가 다수 관찰됐다.

출처: 논문 각 응용 사례(A1·A3·A4) 섹션. 정확한 baseline·설정은 원문 표를 참고.

A2 — Interactive Retrieval 예시

Natural Questions 6,775개 질의 중 약 19.4%가 “UPDATE CONTEXT”를 트리거해 추가 검색을 유도했다. 예: “who carried the usa flag in opening ceremony” — 1차 컨텍스트로는 답을 못 찾다가 컨텍스트를 갱신한 뒤 정답에 도달.

A4 — Safeguard의 역할

Writer가 작성한 코드의 안전성을 Safeguard가 별도로 판정한다. 안전 여부 식별 F1 점수가 GPT-4 기준 +8%p, GPT-3.5 기준 +35%p 개선 — “코드 작성”과 “안전성 검증”을 다른 에이전트로 분리한 효과다.

논문 인용 (A1)

“built-in agents from AutoGen already yield better performance out of the box” — 별도 튜닝 없이 기본 제공 에이전트 조합만으로도 경쟁 방법들을 앞선다는 점을 강조한다.

기존 접근과 무엇이 다른가

CAMEL: 정해진 역할극

두 에이전트가 미리 정한 역할(예: AI User / AI Assistant)로 대화하지만, 도구 사용을 지원하지 않는다. 논문 인용: “CAMEL does not natively support tool usage.”

LangChain: 단일 에이전트 패러다임

대부분의 에이전트가 하나의 실행 루프를 도는 구조라 협업 모드를 염두에 두지 않는다. 논문 인용: “All agents follow single-agent paradigm, not inherently designed for collaborative modes.”

한 줄 정리

다른 프레임워크들은 역할이나 순서를 미리 못박아 두는 반면, AutoGen은 “대화”라는 프로그래밍 가능한 매체 위에서 정적 워크플로우와 즉흥적인 협업을 모두 표현한다.

  • 서로 다른 역할을 분리해 검증·협업시키고 싶을 때 (작성자/검토자 등)
  • 코드 실행 결과를 대화 루프로 자연스럽게 되먹임하고 싶을 때
  • 정적 파이프라인과 동적 그룹챗을 같은 프레임워크로 다루고 싶을 때
  • 완전 자동화 전, human-in-the-loop 단계별 검증이 필요할 때

⚠ 한계 · 유의점

  • 에이전트가 늘수록 “이해 불가능한 잡담”이 될 위험 — “risk resulting in incomprehensible, unintelligible chatter among agents”
  • 높은 자율성은 그 자체로 위험 — “high level of autonomy can pose potential risks”
  • 코드 실행·함수 호출로 외부 환경을 바꾸는 것 자체가 리스크 — “allowing LLM agents to make changes in external environments could be risky”
  • 에이전트 수·역할·상호작용 방식에 대한 “정답 설계”가 없음 — “no one-fits-all answer”
LangGraph로 본다면

AutoGen의 ConversableAgent + GroupChatManager 패턴은 LangGraph의 그래프 구조와 자연스럽게 대응된다 — 에이전트 = 노드, “다음 발화자 선택”(control flow) = 조건부 엣지(conditional edge), 대화 히스토리 = 그래프의 상태(state)로 볼 수 있다. AutoGen이 “대화”라는 런타임 개념으로 표현한 것을, LangGraph는 명시적인 상태 머신으로 컴파일해 더 예측 가능하게 만든 구조라고 이해하면 두 프레임워크의 관계가 선명해진다.
