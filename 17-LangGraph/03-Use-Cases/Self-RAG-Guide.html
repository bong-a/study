Self-RAG

Learning to Retrieve, Generate, and Critique through Self-Reflection
스스로 검색·생성·비평하는 RAG

“필요할 때만 검색하고, 검색한 근거가 맞는지 스스로 채점한다.” 하나의 언어모델이 생성 도중 리플렉션 토큰을 내뱉어 언제 검색할지, 근거가 답을 뒷받침하는지, 답이 쓸모 있는지를 스스로 판단한다.

세 줄 정리

① 문제

일반 RAG는 항상 고정 개수의 문서를 검색해 무조건 집어넣는다. 검색이 불필요하거나 빗나가도 노이즈가 그대로 들어가고, 답이 근거에 충실한지 확인할 장치가 없다.

② 해법

모델이 생성하면서 리플렉션 토큰 4종을 함께 뱉게 학습한다. 필요할 때만 검색(on-demand)하고, 검색한 근거와 자기 출력을 스스로 비평한다.

③ 효과

7B·13B 모델이 ChatGPT·Llama2-chat을 능가. 특히 장문 생성에서 인용(citation) 정확도가 압도적으로 높아 사실성과 출처 신뢰성을 동시에 잡는다.

01 · 동기

왜 “무조건 검색”이 문제인가

RAG는 파라미터 지식만 쓰는 LLM의 환각을 줄여준다. 하지만 대부분의 RAG는 질문 종류와 무관하게 정해진 개수의 문서를 항상 검색해 그대로 이어붙인다. 여기서 세 가지가 어긋난다.

논문 인용

“they indiscriminately retrieve and incorporate a fixed number of retrieved passages, regardless of whether retrieval is necessary or passages are relevant, and do not ensure that generations are consistent with the cited sources.”

불필요한 검색

“여름휴가 에세이를 써줘” 같은 창작 질문엔 검색이 필요 없다. 그런데도 문서를 끌어와 다양성·유창성을 해친다.

관련성 미검증

검색이 빗나가도 문서를 그대로 주입. 무관한 정보가 근거로 둔갑해 환각을 부추긴다.

출처 불일치

생성한 문장이 실제로 인용 문서에 있는지 보장하지 못한다. 그럴듯하지만 근거 없는 답이 나온다.

02 · 핵심 개념

리플렉션 토큰 — 모델이 스스로에게 남기는 4가지 메모

Self-RAG의 심장은 리플렉션 토큰(reflection tokens)이다. 모델의 어휘(vocabulary)에 추가된 특수 토큰으로, 일반 텍스트를 생성하듯 자연스럽게 함께 출력된다. 검색을 제어하는 토큰 1종과, 생성물을 비평하는 토큰 3종으로 나뉜다.

Retrieve지금 검색이 필요한가?

다음 세그먼트를 쓰기 전에 외부 문서를 검색할지 결정한다. 검색을 필요할 때만 부르는 on-demand 검색의 스위치다.

yes · 검색한다 no · 파라미터 지식으로 continue · 기존 근거 계속 사용
IsRel문서가 관련 있나?

검색된 각 문서가 질문에 유용한 정보를 담고 있는지 판정한다. 관련 없으면 그 문서에 기반한 답은 배제된다.

relevant · 관련 있음 irrelevant · 관련 없음
IsSup근거가 답을 뒷받침하나?

생성한 문장이 인용 문서로 실제 검증되는지 3단계로 채점한다. 출처 일치(attribution)를 직접 강제하는 토큰이다.

fully supported partially supported no support / contradictory
IsUse답이 쓸모 있나?

최종 응답이 질문에 얼마나 유용한지를 5점 척도로 평가한다. 근거와 별개로 답 자체의 품질을 본다.

1 2 3 4 5
왜 “토큰”으로 만들었나

판단을 별도 분류기가 아니라 생성 어휘에 녹여 넣었기 때문에, 모델은 텍스트를 쓰는 것과 똑같은 방식으로 비평을 출력한다. 덕분에 추론 시 이 토큰의 확률값을 그대로 읽어 검색 여부·답 선택을 제어할 수 있고, 별도 비평 모델을 돌릴 필요가 없다.

03 · 생성 흐름

한 번의 생성, 실시간 자기 비평

Self-RAG의 출력은 “텍스트 + 리플렉션 토큰”이 뒤섞인 하나의 스트림이다. 아래 테이프처럼 모델은 쓰면서 동시에 스스로를 채점한다. 같은 모델이 질문에 따라 검색을 켜거나 끄는 것이 핵심이다.

이 테이프가 말하는 것

같은 모델이 질문에 맞춰 검색을 켜고 끈다(adaptive retrieval). 검색을 켠 경우엔 문서 관련성(IsRel)과 답–근거 일치(IsSup)까지 스스로 채점하고, 끈 경우엔 유용성(IsUse)만 본다. 비평이 곧 출력의 일부라는 점이 Self-RAG를 이전 RAG와 가르는 지점이다.

04 · 학습

어떻게 이런 모델을 학습하나 — 2단계 증류

핵심 아이디어: 비평 모델(Critic)로 학습 데이터에 리플렉션 토큰을 미리 심어 두고, 그 데이터를 흉내 내도록 생성 모델(Generator)을 학습시킨다. 그러면 추론 때는 생성 모델 혼자 토큰까지 뱉는다.

1단계 · Critic 모델 C

Llama2-7B를 GPT-4의 판단을 증류해 학습. 원본 데이터에서 표본을 뽑아 GPT-4에게 “이 상황에 맞는 리플렉션 토큰을 붙여라”라고 프롬프트하고, 그 라벨을 모방하게 한다.

  • 목표: log pC(r | x, y) 최대화 (r = 리플렉션 토큰)
  • 대부분의 토큰 유형에서 GPT-4와 90%+ 일치
  • 오프라인 데이터 제작용 — 최종 배포엔 쓰지 않는다

2단계 · Generator 모델 M

Critic으로 리플렉션 토큰이 삽입된 코퍼스를 만든 뒤, 그 위에서 일반적인 다음 토큰 예측으로 학습. 텍스트와 토큰을 한꺼번에 예측하도록 배운다.

  • 목표: log pM(y, r | x) — 출력·토큰 동시 예측
  • 검색된 지문은 손실 계산에서 마스킹
  • 어휘에 리플렉션 토큰을 추가해 생성으로 취급

Critic은 데이터 공장, Generator는 완제품. 추론 단계엔 Critic이 사라진다.

왜 이 구조가 영리한가

GPT-4를 매 추론마다 부르면 느리고 비싸다. Self-RAG는 GPT-4의 판단력을 학습 시점에 단 한 번 7B Critic으로 증류하고, 다시 Generator에 이식한다. 그 결과 추론 비용은 일반 LM 한 대 수준이면서, 검색 제어와 자기비평 능력을 내장한다.

05 · 추론

추론 시점 — 검색은 켜고 끄고, 답은 골라 담는다

학습된 Generator는 추론에서 두 개의 조절 손잡이를 준다. 하나는 언제 검색할지(adaptive retrieval), 다른 하나는 여러 후보 중 어떤 세그먼트를 채택할지(segment beam search).

① 적응형 검색 — Retrieve 확률에 임계값

Retrieve=Yes 토큰의 확률이 임계값 δ를 넘으면 검색을 켠다. δ를 낮추면 검색이 잦아져 사실성·인용↑, 높이면 검색이 줄어 유창성·속도↑. 하나의 모델로 재학습 없이 이 균형을 조절할 수 있다.

② 세그먼트 단위 빔서치 — 근거별 후보를 채점

검색을 켜면 K개 문서를 병렬로 처리해 각 문서 기반 후보 세그먼트를 만들고, 아래 점수로 순위를 매겨 상위 후보(빔 폭 2)를 남긴다. 생성 확률에 리플렉션 토큰 점수를 가중합한다.

07 · 비교 분석

Standard RAG vs Self-RAG vs CRAG

“검색 필요 여부와 품질을 어떻게 다루는가”라는 축에서 세 접근을 비교하면 Self-RAG의 자리가 분명해진다. (자매편 CRAG 정리와 함께 보면 좋다.)

항목 Standard RAG Self-RAG CRAG
검색을 언제 하나 ✕ 항상 고정 개수 ✓ 필요할 때만 (Retrieve 토큰) ~ 검색 후 품질로 사후 판단
품질 검사 주체 ✕ 없음 생성기 자신 (reflection token) 외부 경량 평가기 (T5-large 0.77B)
답–근거 일치 강제 ✕ 보장 없음 ✓ IsSup로 직접 채점 ~ strip 정제로 간접
검색 실패 시 대응 ✕ 없음 ~ 무관 문서 배제 (외부 대체는 없음) ✓ 웹 검색으로 대체·보강
생성 LLM 파인튜닝 ✓ 불필요 ✕ 필수 (토큰 학습) ✓ 불필요 (plug&play)
추가 모델 검색기만 학습용 7B Critic (추론엔 불필요) 추론용 0.77B 평가기
조절 손잡이 — 임계값 δ · 가중치 w (재학습 없이) 신뢰도 상·하한 임계값

해석 — 판단을 “안에” 두느냐 “밖에” 두느냐

Self-RAG: 판단을 모델 안에

생성기를 재학습해 검색 제어와 자기비평을 내장한다. 검색을 켜고 끌 수 있고 답–근거 일치를 직접 강제한다. 대신 모델을 바꿔야 하고, 검색이 틀렸을 때 웹 같은 외부 대안이 없다.

CRAG: 판단을 모델 밖에

경량 평가기를 따로 두어 어떤 생성기에도 얹을 수 있다. 검색이 틀리면 웹으로 갈아탄다. 대신 검색을 항상 먼저 수행하고, 성능은 평가기 정확도에 의존한다.

둘은 경쟁이자 상호보완

CRAG 논문은 자신의 교정 모듈을 Self-RAG 생성기 위에 얹은 Self-CRAG로 성능을 더 끌어올렸다. Self-RAG는 “똑똑한 생성기”를, CRAG는 그 앞의 “똑똑한 품질 게이트 + 웹 대체”를 담당한다 — 합치면 가장 강하다.

LangGraph로 구현한다면

Self-RAG는 조건부 그래프로 자연스럽게 매핑된다 — 노드: retrieve_or_not → (조건부) retrieve → grade_relevance(IsRel) → generate → grade_support(IsSup) → grade_usefulness(IsUse). 각 리플렉션 토큰이 곧 등급 판정 노드 + conditional edge가 되어, 지원이 부족하면 재생성·재검색으로 되돌아가는 루프를 만든다. 이 저장소의 17-LangGraph 예제에서 대표적으로 다루는 패턴이다.
