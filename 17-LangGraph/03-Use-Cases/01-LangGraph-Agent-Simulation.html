LangGraph · 03-Use-Cases · 01

가상 고객을 만들어 챗봇을 자동으로 평가하기

상담사 챗봇과 대화하는 "가상 고객(Simulated User)"을 LangGraph 그래프로 만들어, 사람이 매번 손으로 대화하지 않고도 챗봇 품질을 재현 가능하게 테스트하는 패턴입니다.

📓 01-LangGraph-Agent-Simulation.ipynb
왜 필요한가

고객 지원 챗봇처럼 대화형 시스템은 코드를 한 줄 바꿀 때마다 사람이 직접 채팅을 치며 확인해야 해서 테스트가 오래 걸립니다.

해법은 고객 역할을 하는 LLM을 하나 더 만들어 상담사 챗봇과 자동으로 대화를 주고받게 하는 것입니다. LangGraph에서는 이 둘을 각각 노드로 만들고, 대화가 끝나는 조건만 정의하면 됩니다.

01전체 흐름

그래프는 두 노드(상담사·가상 고객)가 서로에게 메시지를 주고받으며 돌다가, 조건을 만족하면 종료됩니다.

사용자 초기 메시지
HumanMessage(content=...)
AI 상담사
ai_assistant_node → call_chatbot()
가상 고객
simulated_user_node → _swap_roles → simulated_user
should_continue(state)
continue
메시지 ≤ 6개 &
마지막 ≠ FINISHED
end
메시지 > 6개 or
마지막 = FINISHED
END
assistant 메시지 추가
user 메시지 추가 (역할 교환 후)
continue — 상담사 노드로 되돌아가 대화 계속 end — 그래프 종료(END)

02핵심 구성요소

그래프를 이루는 6개 조각. 상담사 쪽과 고객 쪽이 대칭 구조로 되어 있습니다.

State 상태 정의

messages 리스트 하나만 있는 단순한 상태입니다. add_messages 리듀서를 붙여서, 노드가 반환한 메시지가 기존 리스트에 자동으로 누적(append)되도록 합니다.

class State(TypedDict):
    messages: Annotated[list, add_messages]
call_chatbot() 상담사

실제 상담사 역할. "항공사 고객 지원 담당자"라는 시스템 프롬프트를 가진 GPT-4 계열 모델을 호출합니다. 이 함수 내부만 바꾸면 단순 LLM 대신 에이전트로도 교체할 수 있습니다.

create_scenario() / simulated_user 고객

가상 고객의 이름(name)과 상황 지시문(instructions)을 받아 고객 역할 프롬프트를 만듭니다. 노트북 예시는 "작년에 다녀온 제주도 여행 전액 환불을 요구하는 Teddy" 시나리오입니다.

시스템 프롬프트에는 대화가 끝나면 'FINISHED' 한 단어로만 응답하라는 규칙이 박혀 있습니다 — 이게 종료 신호로 쓰입니다.

_swap_roles() 역할 교환

상담사와 고객이 둘 다 LLM이라 둘 다 AI 메시지로 응답합니다. 그런데 LangChain 프롬프트는 Human/AI가 번갈아 나오는 걸 기대하므로, 고객 노드를 호출하기 직전에 메시지 역할을 뒤집어(AI↔Human) "고객 입장에서 보는 대화 기록"으로 재구성합니다.

should_continue() 종료 판정

고객 노드 실행 다음에만 호출되는 조건부 엣지 함수. 아래 종료 조건 표의 두 가지 중 하나라도 맞으면 "end", 아니면 "continue"를 반환합니다.

StateGraph 그래프 조립

두 노드를 등록하고, ai_assistant → simulated_user 고정 엣지 하나와, simulated_user 뒤에 should_continue 결과에 따라 갈라지는 조건부 엣지를 연결합니다. 시작점은 ai_assistant입니다.

graph_builder
graph_builder.add_edge("ai_assistant", "simulated_user")
graph_builder.add_conditional_edges(
    "simulated_user", should_continue,
    {"end": END, "continue": "ai_assistant"},
)
graph_builder.set_entry_point("ai_assistant")

03메시지 역할이 왜 뒤집히는가

같은 메시지라도 "누구 입장에서 보느냐"에 따라 역할이 달라집니다. simulated_user_node가 고객을 호출하기 전에 이 변환을 한 번 거칩니다.

상담사 관점 (state 원본)
Human안녕하세요? 화가 많이 났어요
AI어떻게 도와드릴까요?
⇄
고객 관점 (_swap_roles 이후)
AI안녕하세요? 화가 많이 났어요
Human어떻게 도와드릴까요?

즉 상담사가 보낸 말(원래 AI 메시지)이 고객 입장에서는 "상대가 한 말(Human)"이 되고, 고객 자신이 했던 말(원래 Human 메시지)은 "내가 한 말(AI)"이 됩니다.

04대화 종료 조건

should_continue가 매 턴마다 확인하는 두 가지 조건. 둘 중 하나라도 맞으면 즉시 종료합니다.

조건판정 기준의미
길이 초과 len(state["messages"]) > 6 예제를 짧게 유지하기 위한 임의의 상한선. 실제 서비스에선 더 크게 잡을 수 있습니다.
고객이 종료 선언 messages[-1].content == "FINISHED" 가상 고객의 시스템 프롬프트에 박아둔 규칙 — 볼일이 끝났다고 판단하면 이 단어만 출력하도록 지시되어 있습니다.
그 외 "continue" 상담사 노드로 돌아가 대화를 한 턴 더 진행합니다.

05실행 예시

노트북은 "화가 난 고객"의 첫 메시지로 시뮬레이션을 시작합니다. 상담사 노드가 시작점이므로, 이 메시지는 상담사가 받는 최초 입력이 됩니다.

stream_graph 실행
config = RunnableConfig(
    recursion_limit=10,
    configurable={"thread_id": random_uuid()},
)
inputs = {"messages": [HumanMessage(content="안녕하세요? 저 지금 좀 화가 많이 났습니다^^")]}

stream_graph(simulation, inputs, config,
             node_names=["simulated_user", "ai_assistant"])

recursion_limit=10은 그래프가 무한 루프에 빠질 경우를 대비한 안전장치이고, thread_id는 이 대화 세션을 식별하는 값입니다.

·핵심 요약

  • 상담사와 가상 고객을 각각 독립된 LangGraph 노드로 만들면, 둘이 자동으로 대화를 주고받는 루프를 구성할 수 있다.
  • 둘 다 LLM이므로 메시지 역할을 교환하는 로직(_swap_roles)이 반드시 필요하다 — 안 그러면 고객 프롬프트가 "누가 무슨 말을 했는지" 헷갈린다.
  • 종료는 메시지 개수 상한과 고객의 "FINISHED" 선언, 두 조건의 조건부 엣지로 제어한다.
  • 이 패턴을 쓰면 코드 수정마다 사람이 직접 채팅하지 않고도 재현 가능한 챗봇 평가 시나리오를 자동으로 여러 번 돌려 볼 수 있다.
