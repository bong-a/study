CRAG

Corrective Retrieval-Augmented Generation
검색을 교정하는 RAG

“검색이 틀렸을 때 생성까지 무너지지 않도록.” 검색된 문서의 품질을 스스로 평가하고, 필요하면 웹 검색과 노이즈 제거로 지식을 교정한 뒤 답을 만든다.

세 줄 정리

① 문제

일반 RAG는 검색 결과를 무조건 LLM에 넣는다. 검색이 빗나가면 노이즈가 그대로 들어가 환각(hallucination)을 유발한다.

② 해법

경량 검색 평가기가 문서 관련도를 채점 → 신뢰도에 따라 3가지 교정 액션(사용 / 웹검색 / 혼합)을 자동 선택.

③ 효과

생성 LLM을 건드리지 않는 plug-and-play. Self-RAG에 얹은 Self-CRAG가 4개 벤치마크 대부분에서 성능을 끌어올림.

01 · 동기

왜 일반 RAG는 위험한가

RAG의 성능은 검색기(retriever)의 정확도에 절대적으로 의존한다. 그런데 대부분의 RAG는 검색된 문서가 실제로 관련이 있든 없든 가리지 않고 그대로 생성기에 밀어 넣는다.

논문 인용

“a low-quality retriever is prone to introducing a substantial amount of irrelevant information, impeding the generators from acquiring accurate knowledge and potentially misleading them.”

“most conventional RAG approaches indiscriminately incorporate the retrieved documents, regardless of whether these documents are relevant or not.”

문제 지점: 검색이 틀려도 이를 감지·교정할 장치가 전혀 없다.

02 · 개요

CRAG 파이프라인 한눈에

CRAG는 검색과 생성 사이에 ‘교정 계층(corrective layer)’을 끼워 넣는다. 검색 결과를 평가하고, 그 결과에 따라 지식을 다듬거나 웹에서 보강한 뒤 생성기로 넘긴다.

3~5번 단계가 CRAG의 핵심 — 기존 RAG에 그대로 끼워 넣을 수 있는 모듈이다.

핵심 아이디어

“검색이 맞았는지 따로 검사하고, 틀렸으면 다른 지식원(웹)으로 갈아탄다. 맞았어도 문서 속 불필요한 부분은 잘라낸다.” — 이 세 가지가 CRAG를 이룬다.

03 · 검색 평가기

Retrieval Evaluator — 관련도를 채점하는 경량 심판

CRAG는 T5-large를 파인튜닝한 경량 평가기를 둔다. (질문, 문서) 쌍마다 관련도 점수를 예측하고, 이를 모아 최종 신뢰도(confidence)를 산출한다.

0.77B
CRAG 평가기 (T5-large)
생성기와 별개로 동작하는 외부 모듈
≈ 9×
Self-RAG 대비 가벼움
Self-RAG의 critic은 7B 규모
쌍-단위
문서별 개별 채점
질문+문서 concat → 관련도 예측

동작 방식

  • 검색된 각 문서를 질문과 짝지어 관련도 점수를 개별 예측한다.
  • 문서별 점수를 종합해 하나의 최종 판정(신뢰도)으로 정리한다.
  • 이 신뢰도를 상·하한 임계값과 비교해 다음 단계(3-액션)를 결정한다.

※ 별도의 대형 비평 모델 없이 0.77B 평가기 하나만 추가하므로, 생성 LLM을 재학습할 필요가 없다(= plug-and-play의 근거).

04 · 핵심

신뢰도로 갈리는 3가지 교정 액션

평가기의 신뢰도를 하한·상한 두 임계값으로 잘라 세 구간으로 나눈다. 구간마다 사용할 지식원과 처리 방식이 달라진다.

✓ CORRECT

관련 있음

조건: 최소 한 문서의 신뢰도가 상한 임계값 초과.
처리: 검색 문서에 지식 정제(decompose-then-recompose)를 적용해 핵심만 남겨 사용.

~ AMBIGUOUS

애매함

조건: 상·하한 사이 — 확신이 서지 않는 경우.
처리: Correct의 내부 정제 지식과 Incorrect의 웹 검색 지식을 함께 사용해 서로 보완.

✕ INCORRECT

관련 없음

조건: 모든 문서의 신뢰도가 하한 임계값 미만.
처리: 검색 문서를 전부 폐기하고, 웹 검색으로 외부 지식을 새로 확보.

※ 임계값은 고정 상수가 아니라 데이터셋별로 실험적으로 튜닝한다. 그림의 구간 비율은 이해를 돕기 위한 예시다.

06 · 웹 검색 교정

검색이 실패하면 웹으로 갈아탄다

Incorrect·Ambiguous 상황에서 CRAG는 대규모 웹 검색으로 외부 지식을 끌어온다. 정적인 코퍼스의 한계(오래됨·좁음)를 열린 웹으로 보완하는 것이 핵심이다.

읽는 법 ①

강한 생성기에서 이득이 크다. Self-CRAG는 PopQA +6.9, Bio +5.0, PubHealth +2.4로 Self-RAG를 확실히 앞선다.

읽는 법 ②

약한 생성기에선 이득이 작다. LLaMA2-hf-7b는 지시를 잘 못 따라(PubHealth·ARC가 바닥), 좋은 지식을 줘도 활용하지 못한다.

→ 결론: CRAG의 교정이 빛을 보려면 지식을 제대로 소화할 수 있는 생성기가 전제되어야 한다. 교정은 “좋은 재료”를 주는 것이고, 요리는 생성기 몫이다.

08 · 비교 분석

Standard RAG vs Self-RAG vs CRAG

세 접근을 “검색 실패에 어떻게 대응하는가”라는 축에서 비교하면 CRAG의 자리가 분명해진다.

항목 Standard RAG Self-RAG CRAG
검색 품질 검사 ✕ 없음 (무조건 사용) ✓ 생성기가 낸 reflection token으로 자기 판단 ✓ 외부 경량 평가기(T5)가 채점
검색 실패 시 교정 ✕ 없음 ~ 무시/비판은 하나 외부 대체 없음 ✓ 웹 검색으로 지식 대체·보강
외부(웹) 지식 ✕ 코퍼스 한정 ✕ 코퍼스 한정 ✓ 열린 웹 검색
문서 내 노이즈 처리 ✕ 통째로 주입 ~ 세그먼트 관련성 비판 ✓ strip 단위 분해·필터
생성 LLM 파인튜닝 ✓ 불필요 ✕ 필수 (reflection token 학습) ✓ 불필요 (plug&play)
추가 모델 검색기만 7B critic으로 학습 데이터 생성 → 생성기 재학습 0.77B 평가기만 추가
기존 RAG에 결합 — ~ 구조 변경 필요 ✓ 그대로 얹기 (Self-CRAG)

해석 — 무엇이 다른가

Self-RAG: 생성기가 스스로 판단

생성 LLM을 재학습해 Retrieve / IsRel / IsSup / IsUse 같은 reflection token을 뱉게 만든다. 판단이 모델 내부에 녹아 강력하지만, 생성기를 바꿔야 하고 검색이 틀렸을 때 외부 대안이 없다.

CRAG: 판단을 밖으로 분리

판단을 0.77B 외부 평가기로 떼어내 생성기를 건드리지 않는다. 그래서 어떤 RAG/LLM에도 끼워 넣을 수 있고, 검색이 틀리면 웹으로 갈아탄다. 대신 성능은 평가기 정확도에 의존한다.

한 줄 정리

Self-RAG는 “똑똑한 생성기”를 만들고, CRAG는 생성기 앞에 “똑똑한 품질관리 게이트”를 세운다. 둘은 경쟁이자 상호보완이라 합치면(Self-CRAG) 가장 강하다.

09 · 정리

정리 · 언제 쓰나 · 한계

✓ 이럴 때 쓰면 좋다

  • 검색 코퍼스가 불완전·구식일 수 있어 웹 보강이 필요할 때
  • 생성 LLM을 재학습하기 어려운 환경(플러그인식 도입)
  • 검색 노이즈로 인한 환각을 억제하고 싶을 때
  • 이미 쓰는 RAG 파이프라인에 모듈만 얹고 싶을 때

⚠ 한계 · 유의점

  • 성능이 평가기 정확도에 좌우된다(잘못 채점하면 교정도 어긋남)
  • 웹 검색은 지연·신뢰성·비용 이슈가 따른다
  • 임계값을 데이터셋마다 튜닝해야 한다
  • 약한 생성기에서는 이득이 제한적(실험에서 확인)
LangGraph로 구현한다면

CRAG는 상태 머신(state machine)으로 아주 자연스럽게 매핑된다 — 노드: retrieve → grade_documents → (조건부) → generate 또는 transform_query → web_search → generate. 평가기의 3-액션이 곧 conditional edge가 된다. 이 저장소의 17-LangGraph 패턴과 잘 맞아, “Corrective RAG”는 대표적인 LangGraph 예제로 자주 구현된다.
