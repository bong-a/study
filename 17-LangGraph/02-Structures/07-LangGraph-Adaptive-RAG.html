Adaptive RAG

쿼리를 분석해 최적의 검색 경로를 선택하고, 생성된 답변을 스스로 검증·수정하는 자기 수정형 검색 증강 생성 시스템.

쿼리 라우팅
문서 관련성 평가
환각 탐지
쿼리 재작성
자기 수정 루프

Adaptive RAG란?

기존 RAG의 한계를 극복하기 위해 두 가지 전략을 결합합니다.

🗺️

쿼리 분석 (Query Analysis)

질문의 성격을 파악해 어디서 검색할지를 먼저 결정합니다. PDF 지식베이스에 관한 질문이면 벡터스토어로, 최신 뉴스나 실시간 정보가 필요하면 웹 검색으로 라우팅합니다.

🔄

Self-Reflective RAG

검색 결과와 생성된 답변을 LLM이 직접 평가합니다. 관련 없는 문서는 걸러내고, 환각이 포함된 답변은 재생성하며, 답변이 부정확하면 쿼리를 다시 작성해 재검색합니다.

🎯

기존 RAG와의 차이

기본 RAG는 "검색 → 생성" 한 번으로 끝납니다. Adaptive RAG는 결과가 불만족스러우면 스스로 원인을 파악하고 경로를 수정하며, 만족할 때까지 반복합니다.

전체 워크플로우

LangGraph로 구성한 Adaptive RAG의 노드와 엣지 흐름. 조건에 따라 루프가 발생할 수 있습니다.

조건 분기 (Decision)
웹 검색
문서 검색
쿼리 재작성
답변 생성
재시도 루프

핵심 컴포넌트

Adaptive RAG를 구성하는 6개의 핵심 모듈. 각각 독립적으로 동작하며 그래프 노드로 연결됩니다.

question_router
라우터

질문을 분석해 벡터스토어 또는 웹 검색 중 어디로 보낼지 결정합니다. LLM이 구조화된 출력으로 데이터소스를 선택합니다.

class RouteQuery(BaseModel):
    datasource: Literal[
        "vectorstore",
        "web_search"
    ]

# 구조화 출력 강제
router = prompt | llm.with_structured_output(
    RouteQuery
)

예시:
"삼성 가우스의 이름은?" → vectorstore
"판교 딤섬집 추천" → web_search

retrieval_grader
평가기

검색된 각 문서가 질문과 관련이 있는지 yes/no로 평가합니다. 무관한 문서는 필터링해 노이즈를 제거합니다.

class GradeDocuments(BaseModel):
    binary_score: str  # "yes" | "no"

# 각 문서에 대해 평가
for doc in documents:
    score = grader.invoke({
        "question": question,
        "document": doc.page_content
    })
    if score.binary_score == "yes":
        filtered.append(doc)
rag_chain
생성기

필터링된 문서(컨텍스트)를 바탕으로 최종 답변을 생성합니다. LangChain Hub의 RAG 프롬프트를 사용합니다.

# Hub에서 RAG 프롬프트 로드
prompt = hub.pull("teddynote/rag-prompt")

rag_chain = prompt | llm | StrOutputParser()

# 생성 실행
generation = rag_chain.invoke({
    "context": format_docs(docs),
    "question": question
})
hallucination_grader
검증기

생성된 답변이 실제 문서에 근거하는지(grounded) 확인합니다. 환각이 있으면 답변 재생성을 지시합니다.

class GradeHallucinations(BaseModel):
    binary_score: str  # "yes"=근거있음

score = grader.invoke({
    "documents": docs,
    "generation": generation
})
# "yes" → 다음 단계로
# "no"  → generate 재실행
answer_grader
검증기

답변이 질문을 실제로 해결하는지 평가합니다. 근거는 있지만 질문과 엉뚱한 답변이면 쿼리 재작성으로 보냅니다.

class GradeAnswer(BaseModel):
    binary_score: str  # "yes" | "no"

# hallucination 통과 후 실행
score = grader.invoke({
    "question": question,
    "generation": generation
})
# "yes" → END (완료)
# "no"  → transform_query
question_rewriter
재작성기

벡터스토어 검색에 최적화된 형태로 질문을 개선합니다. 검색 실패 시 자동으로 호출되어 재검색합니다.

system = """질문을 벡터스토어 검색에
최적화된 형태로 변환하세요.
의미적 의도를 파악하여
더 나은 검색어를 생성합니다."""

rewriter = re_write_prompt | llm | StrOutputParser()

# "삼성AI이름?"
# → "삼성전자가 개발한 생성형 AI 모델명"

그래프 구성 코드

LangGraph의 StateGraph로 노드와 엣지를 연결해 전체 워크플로우를 정의합니다.

compile_graph.py 
workflow = StateGraph(GraphState)

# 노드 등록
workflow.add_node("web_search",     web_search)
workflow.add_node("retrieve",       retrieve)
workflow.add_node("grade_documents", grade_documents)
workflow.add_node("generate",       generate)
workflow.add_node("transform_query", transform_query)

# START → 라우팅 조건 분기
workflow.add_conditional_edges(
    START, route_question,
    {"web_search": "web_search", "vectorstore": "retrieve"}
)

# 단순 엣지
workflow.add_edge("web_search",     "generate")
workflow.add_edge("retrieve",       "grade_documents")
workflow.add_edge("transform_query", "retrieve")

# 문서 관련성 조건 분기
workflow.add_conditional_edges(
    "grade_documents", decide_to_generate,
    {"transform_query": "transform_query", "generate": "generate"}
)

# 환각 체크 조건 분기 (3가지 출구)
workflow.add_conditional_edges(
    "generate", hallucination_check,
    {
        "hallucination":  "generate",        # 재생성
        "relevant":       END,               # 완료
        "not relevant":   "transform_query", # 재시도
    }
)

app = workflow.compile(checkpointer=MemorySaver())
GraphState — 공유 상태 객체
class GraphState(TypedDict):
    question:   Annotated[str,       "사용자 질문"]
    generation: Annotated[str,       "LLM 생성 답변"]
    documents:  Annotated[List[str], "검색/필터링된 문서 목록"]

모든 노드는 이 상태를 읽고 업데이트합니다. 노드 함수는 변경할 키만 반환하면 되며, LangGraph가 병합을 처리합니다.

RAG 전략 비교

Adaptive RAG 논문에서 제시하는 세 가지 전략의 차이.

전략 경로 속도 정확도 사용 케이스
No Retrieval LLM 직접 응답 빠름 낮음 일반 상식, 간단한 사실
Single-shot RAG 검색 → 생성 (1회) 보통 보통 명확한 질문, 관련 문서 풍부
Adaptive RAG 라우팅 → 검색/웹 → 평가 → 재시도 느릴 수 있음 높음 복잡한 질문, 최신 정보 혼재
장점
  • ✓질문 유형에 따라 최적 경로 자동 선택
  • ✓환각 발생 시 자동 감지 후 재생성
  • ✓관련 없는 문서는 자동으로 필터링
  • ✓최신 정보와 문서 기반 정보를 모두 처리
고려사항
  • !LLM 호출 횟수 증가 → 비용·지연 상승
  • !루프가 반복되면 recursion_limit 설정 필요
  • !라우터 오판 시 불필요한 웹 검색 발생 가능
  • !평가 프롬프트 품질이 전체 정확도에 영향

Agentic RAG vs Adaptive RAG

개념적으로 Agentic이 더 넓은 범주입니다. 그리고 Adaptive가 "Adaptive"인 이유가 있습니다.

AGENTIC RAG (상위 개념)
LLM이 에이전트로서 도구를 자율적으로 선택·실행.
흐름이 열려 있고, 모델이 판단.
ADAPTIVE RAG (하위 구현)
Agentic의 동작을 명시적 그래프로 구조화한 것.
라우팅·평가·재시도 경로가 사전에 정의됨.
NAIVE RAG (기본)
검색 → 생성 (1회, 고정 경로)
Tool Use Agent
ReAct Agent
Plan-and-Execute
Adaptive RAG ↑ 포함
관점 Agentic RAG Adaptive RAG
제어 흐름 모델이 즉흥적으로 결정 (자유) 그래프로 경로가 미리 정의 (구조적)
예측 가능성 낮음 — 같은 질문에 다른 경로 높음 — 조건만 같으면 동일 경로
디버깅 어려움 (모델 내부 결정) 쉬움 (엣지·노드 추적 가능)
루프 제어 모델 판단 (무한루프 위험) recursion_limit으로 명시 제한
확장성 새 도구 추가 시 프롬프트만 수정 새 노드·엣지 추가로 그래프 확장
비용 예측 어려움 — LLM 호출 횟수 가변 보통 — 경로별 최대 호출 횟수 추정 가능
Q. 왜 "Adaptive"인가?

논문의 핵심 아이디어는 질문의 복잡도에 따라 검색 전략 자체를 바꾼다는 것입니다.
단순한 질문에 무거운 반복 검색을 쓰는 건 낭비고, 복잡한 질문에 단발 검색은 부족합니다.

🟢
단순 질문
검색 없이 LLM 직접 답변
"파이썬이 뭐야?"
🟡
중간 복잡도
한 번 검색 후 생성
"삼성 가우스 이름은?"
🔴
복잡한 질문
평가·재시도 반복
"AI 규제 국제 동향은?"

검색 깊이(depth)와 경로(path)가 질문에 적응(adapt)하는 것 — 이것이 이름의 이유입니다.
이 LangGraph 구현에서는 웹 검색 ↔ 벡터스토어 라우팅과 자기수정 루프가 그 역할을 합니다.
