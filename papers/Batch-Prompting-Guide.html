30초 요약

Batch Prompting은 모델이나 API를 바꾸는 기술이 아닙니다. 프롬프트 포맷만 바꿔서 비용이 큰 공유 부분을 한 번만 보내고, 여러 샘플의 답을 같이 생성하게 합니다.

N/b API 호출 수 N개 샘플을 b개씩 묶으면 호출 수가 N에서 N/b로 감소합니다.
5x 최대 비용·시간 절감 논문은 b=6 조건에서 최대 5배 수준의 토큰·시간 효율 개선을 보고합니다.
10 평가 데이터셋 상식 QA, 산술 추론, NLI/NLU 범주의 10개 데이터셋에서 검증했습니다.
2-4 실무 권장 시작점 b가 너무 커지면 혼동이 늘어납니다. 작은 배치부터 검증하는 편이 안전합니다.
Bottom line

프롬프트의 대부분이 공통 지시문·few-shot 예시라면 Batch Prompting은 매우 현실적인 절감법입니다. 반대로 입력 하나가 길거나 답변 자체가 길고 창의적인 작업이면 성능 저하와 파싱 실패가 커질 수 있습니다.

핵심 아이디어

표준 프롬프팅은 하나의 입력마다 같은 예시를 다시 보냅니다. Batch Prompting은 예시도 배치 형태로 보여주고, 테스트 입력도 Q[1], Q[2]처럼 인덱스를 붙여 한 번에 묶습니다.

표준 프롬프팅

샘플 하나마다 전체 few-shot 프롬프트를 반복합니다.

call 1
예시예시Q1A1
call 2
예시예시Q2A2
call 3
예시예시Q3A3

같은 예시 토큰을 계속 지불합니다.

Batch Prompting

여러 입력과 여러 답변을 인덱스로 대응시켜 한 번에 처리합니다.

call 1
예시[1]예시[2] Q[1]Q[2]Q[3] A[1]A[2]A[3]

공통 예시와 지시문을 여러 샘플이 나눠 쓰게 됩니다.

왜 비용이 줄어드는가

few-shot 프롬프팅에서는 매 요청에 들어가는 예시와 지시문이 길고, 실제 새 질문은 상대적으로 짧은 경우가 많습니다. Batch Prompting은 이 공유 부분을 배치 안의 여러 샘플이 함께 쓰도록 만듭니다.

논문의 비용 직관

N은 전체 샘플 수, b는 한 번에 묶는 샘플 수, K는 few-shot 예시 수입니다.

호출 수: N → N / b 표준 토큰 효율 η = 1 / (K + 1) 배치 토큰 효율 η = b / (K + b)

K가 충분히 크고 b가 너무 크지 않으면, 샘플당 공유 프롬프트 비용이 거의 b분의 1로 줄어듭니다.

K=12일 때 토큰 효율 예시

논문 실험은 주로 12-shot 예시를 사용했습니다. 아래는 η = b/(12+b)로 본 직관입니다.

주의

시간 절감은 API 호출 오버헤드와 rate limit이 병목일 때 가장 큽니다. 자체 서빙처럼 호출 오버헤드가 작다면, 긴 출력을 한 번에 생성하는 디코딩 비용이 커져 시간 이득이 줄어들 수 있습니다.

어디서 실패하기 쉬운가

Batch Prompting은 비용 절감 기술이지, 정확도를 항상 올리는 기술은 아닙니다. 논문 분석에서 반복적으로 나온 위험 신호는 배치 크기, 입력 길이, 작업 복잡도입니다.

잘 맞는 상황

  • ✓동일한 지시문·예시를 많은 입력에 반복 적용한다.
  • ✓각 입력이 짧고 출력도 짧다. 예: 분류, 단답 QA, 간단 추론.
  • ✓결과를 인덱스나 JSON으로 안정적으로 파싱할 수 있다.
  • ✓API 호출 오버헤드나 rate limit이 실제 병목이다.

주의할 상황

  • !입력 컨텍스트가 길다. WikiTQ 실험에서 긴 테이블 입력은 b가 커질수록 하락했습니다.
  • !창작문, 긴 설명 생성처럼 출력이 긴 작업이다. 공유 프롬프트 절감 효과가 작아집니다.
  • !문제 난도가 높아 모델이 다른 샘플의 내용에 쉽게 방해받는다.
  • !b=6 이상으로 키운다. 논문에서는 b가 커질수록 정확도 저하가 뚜렷했습니다.

실무 적용 레시피

실서비스에 바로 넣기보다는 작은 샘플로 비용, 파싱 성공률, 품질 하락폭을 같이 측정하는 방식이 좋습니다.

1

작게 시작

기존 단건 프롬프트와 b=2, b=3, b=4를 같은 샘플 100-300개에서 비교합니다.

2

출력 형식 고정

인덱스 누락, 순서 뒤섞임, 여분 설명을 막기 위해 JSON 배열이나 번호별 블록을 강하게 요구합니다.

3

재시도 정책 추가

특정 인덱스가 빠지면 빠진 항목만 단건 프롬프트로 재시도하는 fallback을 둡니다.

System:
You answer each item independently. Return only valid JSON.

User:
Classify each question. Keep the same id.

Examples:
Input[1]: ...
Input[2]: ...
Output:
[
  {"id": 1, "label": "...", "reason": "..."},
  {"id": 2, "label": "...", "reason": "..."}
]

Now classify:
Input[1]: ...
Input[2]: ...
Input[3]: ...
Input[4]: ...

Return exactly 4 JSON objects, one per id.
추천 판단

대량 분류, 상품 속성 추출, 짧은 QA 평가, 로그/리뷰 라벨링처럼 같은 프롬프트를 많이 반복하는 업무라면 먼저 b=2 또는 b=3으로 A/B 테스트할 가치가 큽니다. 생성형 장문 작업에는 기본값으로 쓰지 않는 편이 낫습니다.
