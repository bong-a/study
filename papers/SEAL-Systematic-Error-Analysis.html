Paper Brief · arXiv:2408.10270

SEAL: 보상 모델이 정말 사람의 선호를 배웠는지 해부하는 방법

이 논문은 RLHF의 보상 모델을 단순 정확도 대신 어떤 특징에 보상을 주는지, 인간 선호 데이터와 얼마나 자주 어긋나는지, 표현이 바뀌어도 판단이 안정적인지로 분석한다. 핵심 결론은 간단하다. 보상 모델은 선호 데이터의 많은 신호를 배우지만, 스타일이나 안전 관련 표면 특징에도 꽤 민감하게 흔들린다.

1 · 문제의식

RLHF에서 보상 모델은 작은 판사다. 문제는 판사의 기준이 잘 안 보인다는 것.

RLHF에서는 사람의 선호 데이터를 바탕으로 보상 모델을 만들고, 이 보상 모델이 언어 모델의 출력을 점수화한다. 그런데 보상 모델이 높은 점수를 준 이유가 정말 도움됨과 무해함 때문인지, 아니면 문체, 거절 문구, 길이, 긍정적 말투 같은 스포일러 특징 때문인지 알기 어렵다. SEAL은 이 블랙박스를 feature-level로 열어 보는 분석 절차다.

RLHF pipeline with SEAL audit points 사람 선호 → 보상 모델 → 정책 업데이트
Target features

정말 배우고 싶은 가치

논문 실험에서는 helpful과 harmless가 대표 target이다. 사람이 더 선호한 답변이 보통 더 도움되고 덜 해롭다는 전제를 둔다.

Spoiler features

겉으로 그럴듯한 신호

웅변적 문체, 긍정적 감정, 거절 표현, 개인정보 언급 회피 같은 특징이다. 유용할 때도 있지만 target과 완전히 같은 것은 아니다.

Main risk

좋은 점수의 이유가 틀릴 수 있음

RM이 chosen을 맞히더라도, 실제로는 helpfulness가 아니라 특정 말투를 보상했을 가능성을 분리해 봐야 한다.

2 · SEAL 방법

SEAL은 데이터, 보상 모델, 재작성 반응을 순서대로 검사한다.

논문은 Anthropic HH-RLHF 데이터에서 각 프롬프트별 chosen/rejected 응답을 가져오고, LM-labeler로 응답 특징을 태깅한 뒤, pre-D와 post-D 보상 모델의 점수를 비교한다. 마지막으로 일부 응답을 Mistral로 다시 써서, 의미는 비슷한데 스타일이 달라졌을 때 RM 판단이 얼마나 바뀌는지 본다.

SEAL analysis loop feature labels + reward scores + perturbation

Target features helpful, harmless 보상 모델이 실제로 배워야 하는 가치 chosen/rejected 차이가 커야 데이터가 명확하다.
Spoiler features eloquent, sentiment, refusal, privacy, explicitness 등 17개 target과 섞여 보상에 영향을 줄 수 있는 표면 신호 스포일러가 강하면 보상 해석이 어려워진다.
Reward models pre-D RM과 post-D RM 선호 데이터 학습 전후의 판단 비교 post-D가 D에 맞춰졌는지, 원래 맞던 판단이 깨졌는지 본다.
Perturbation 1% subset을 Mistral-7B-Instruct-v0.2로 rewrite 문체/표현을 바꾸어 보상 판단 안정성 측정 의미 유사도와 feature flip을 같이 본다.
3 · 세 가지 지표

이 논문을 이해하려면 세 단어만 잡으면 된다: imprint, resistance, robustness.

SEAL의 장점은 "보상 모델 정확도"라는 한 숫자를 쪼갠다는 점이다. 어떤 특징이 점수에 찍혔는지, 데이터의 선호와 얼마나 어긋나는지, 그리고 작은 표현 변화에 얼마나 취약한지를 분리한다.

Feature imprint

특징 하나가 보상 점수를 얼마나 움직이나

예를 들어 harmless 태그가 붙은 답변에 RM이 평균적으로 더 높은 점수를 주면 harmless imprint가 양수다.

reward = prompt fixed effect + Σ β(feature)
Alignment resistance

chosen을 rejected보다 높게 주지 못하는 비율

human preference pair에서 RM(chosen) ≤ RM(rejected)이면 그 쌍은 misaligned다.

resistance = 1 - P[ RM(chosen) > RM(rejected) ]
Alignment robustness

말투를 바꿔도 판단이 유지되는가

응답을 rewrite했을 때 feature flip이 reward misalignment를 유발하는지 로지스틱 회귀로 본다.

misalignment ~ feature flips + similarity controls
직관: SEAL은 보상 모델을 "답을 맞혔나?"가 아니라 "왜 맞혔나?", "어디서 틀렸나?", "겉모습이 바뀌면 아직 맞히나?"로 나누어 검사한다.
4 · 핵심 결과

post-D 모델은 확실히 D에 맞춰졌지만, 오류가 사라진 것은 아니다.

가장 중요한 숫자는 post-D alignment score 0.74다. 즉, 분석한 선호 쌍의 74%에서는 post-D RM이 chosen에 rejected보다 높은 점수를 줬다. 하지만 남은 26%는 여전히 D와 어긋나며, 특히 pre-D에서는 맞았는데 post-D에서 틀린 8% 역전도 관찰된다.

Feature imprint shift

harmlessness 신호가 강하게 각인됨

harmless feature의 회귀 계수는 pre-D -0.85에서 post-D 2.09로 이동했다. helpful feature도 -0.01에서 1.08로 커졌다. 선호 데이터 D가 RM의 보상 지형을 target 방향으로 크게 밀었다는 뜻이다.

Label ambiguity

73%는 target feature만으로 구분하기 어려움

LM-labeler는 대부분의 response pair에서 chosen과 rejected를 target feature만으로 명확히 구분하지 못했다. 그래서 데이터는 "명확한 가치 차이"뿐 아니라 길이, 표현 방식, 안전 문구 같은 혼합 신호를 함께 담고 있다.

Robustness after rewrite

표현을 바꾸면 정렬 점수가 조금 떨어진다

rewrite subset에서 alignment score는 post-D 기준 0.71로 측정됐다. 원 데이터의 0.74보다 낮지만, 논문은 confidence interval을 고려하면 큰 차이는 아니라고 해석한다.

Important flip

sentiment 변화가 misalignment와 연결됨

rewrite가 sentiment를 바꾼 경우 reward misalignment 가능성이 유의하게 증가했다. 논문은 이를 RM이 감정적 톤 같은 표면 특징에 민감할 수 있다는 증거로 본다.

5 · 해석

이 논문의 메시지는 "보상 모델을 feature 단위로 감사하라"에 가깝다.

For researchers

정확도 하나로는 부족하다

정렬 평가에는 RM accuracy뿐 아니라 어떤 feature가 보상에 각인됐는지, 오류가 어떤 feature 주변에 몰리는지 봐야 한다.

For dataset builders

선호 데이터의 혼합 신호를 줄여야 한다

chosen이 더 안전하면서 동시에 더 장황하고 긍정적이면 RM은 무엇을 배워야 하는지 헷갈린다. 데이터 설계가 중요하다.

For model evaluators

perturbation은 실전형 테스트다

의미는 유지하고 표면 표현을 바꿔 보상 판단이 유지되는지 확인하면 reward hacking 취약성을 더 빨리 발견할 수 있다.

한 줄 요약: SEAL은 RLHF 보상 모델이 "사람이 고른 답"을 얼마나 맞히는지에서 멈추지 않고, 그 답을 고른 이유가 가치 신호인지 겉보기 신호인지 분해하는 감사 도구다.
6 · 한계와 주의점

SEAL도 기준점과 라벨러에 의존한다. 그래서 결과는 감사 신호로 읽어야 한다.

  • 선호 데이터 D를 기준으로 삼는다. D 자체가 편향되거나 불완전하면 SEAL은 그 불완전성도 함께 드러내지만, 최종 가치 판단을 자동으로 해결하지는 않는다.
  • LM-labeler의 feature 라벨에 의존한다. GPT-4-turbo가 붙인 라벨이 taxonomy 밖의 중요한 현상을 놓칠 수 있다.
  • 실험은 특정 데이터와 보상 모델 조합의 시연이다. 모든 RLHF 보상 모델에 같은 수치가 나온다는 일반화가 아니다.
  • spoiler taxonomy는 설계자의 선택이다. 의료, 법률, 교육처럼 도메인이 바뀌면 feature 목록도 다시 만들어야 한다.
  • rewrite 테스트는 의미 보존이 완벽하지 않다. 논문은 유사도를 통제하지만, 작은 의미 변화가 완전히 제거되었다고 볼 수는 없다.
