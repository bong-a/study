STORM 논문 정리

LLM에게 “글을 바로 쓰게” 하지 않고, 먼저 관점별 질문으로 조사하고 개요를 만드는 시스템

핵심은 생성 모델의 문장력보다 사전 조사(pre-writing)를 자동화하는 데 있다. STORM은 관련 문서를 훑어 관점을 만들고, 각 관점의 가상 작성자가 전문가에게 질문하며, 검색 근거를 모아 글의 뼈대와 참고자료를 만든다.

Synthesis Topic outlines Outline-first writing Retrieval grounded Multi-perspective Q&A
Assisting in Writing Wikipedia-like Articles From Scratch with Large Language Models
Authors
Yijia Shao, Yucheng Jiang, Theodore A. Kanell, Peter Xu, Omar Khattab, Monica S. Lam
Goal
주어진 주제만으로 위키백과식 장문 글을 쓰기 위한 조사, 개요, 인용 기반 작성 파이프라인 제안
Dataset
FreshWiki: 최근 생성/대폭 수정된 고품질 Wikipedia 문서로 구성해 학습 데이터 누수를 줄인 평가셋
Takeaway
더 넓고 조직적인 글을 만들지만, 중립성·검증가능성·출처 편향은 여전히 남는다.
01
Big Picture

STORM이 풀려는 문제

일반 RAG는 “주제 검색 결과를 넣고 답을 쓰기”에 가깝다. STORM은 장문 설명문을 쓰기 전에 사람이 하듯이 어떤 관점에서 무엇을 조사할지 먼저 정한다.

문제

긴 글은 바로 쓰기 어렵다

Wikipedia식 글은 사실 나열이 아니라 주제의 범위, 균형, 구조, 출처가 맞아야 한다. 그래서 작성 전 조사와 개요 설계가 중요하다.

관찰

직접 질문은 얕아진다

LLM에게 “질문 30개를 만들어라”라고 하면 대개 언제, 어디서, 무엇 같은 표면 질문에 머문다.

해법

역할과 대화를 준다

이벤트 기획자, 역사 연구자, 경제 분석가처럼 서로 다른 관점의 작성자를 만들면 더 깊고 다양한 질문이 나온다.

한 문장 요약 STORM은 “검색으로 자료를 찾는 시스템”이라기보다 “좋은 검색 질문을 만들기 위해 여러 관점의 인터뷰를 시뮬레이션하는 시스템”이다.
02
Method

STORM 파이프라인

논문은 전체 작업을 사전 조사와 작성으로 나눈다. STORM의 차별점은 사전 조사 단계에서 관점 발견, 다중 턴 질문, 신뢰 가능한 검색 근거 수집을 결합한다는 점이다.

STEP 1

관련 문서 훑기

주제와 비슷한 Wikipedia 문서의 목차를 모아, 어떤 관점이 필요한지 힌트를 얻는다.

STEP 2

관점 생성

기본 사실 작성자에 더해 역사, 기술, 정책, 영향 등 서로 다른 역할을 만든다.

STEP 3

관점별 인터뷰

각 작성자가 전문가에게 질문한다. 답변을 읽고 후속 질문을 이어가며 조사를 깊게 만든다.

STEP 4

검색과 선별

복잡한 질문을 검색 쿼리로 나누고, 신뢰도가 낮은 출처를 거른 뒤 답변을 합성한다.

STEP 5

대화 기록 수집

질문, 답변, 근거 문서를 관점별로 모아 참고자료 집합과 조사 메모를 만든다.

STEP 6

초안 개요 생성

LLM의 내부 지식으로 먼저 일반적인 개요를 만들고 전체 구조의 출발점으로 삼는다.

STEP 7

개요 정제

관점별 대화와 초안 개요를 함께 넣어 빠진 세부 주제와 섹션 구조를 보완한다.

STEP 8

섹션별 작성

개요의 각 섹션에 맞는 참고자료를 검색해 인용이 달린 장문 글을 만든 뒤 중복을 다듬는다.

Question Loop

질문 하나가 다음 질문을 만든다

질문
관점과 대화 기록 기반
→
검색·선별
쿼리 분해, 신뢰 출처 필터
→
답변 합성
근거를 읽고 후속 질문 준비

단발 질문 목록보다 강한 이유는 답변을 읽은 뒤 모르는 지점을 다시 물어볼 수 있기 때문이다.

Why Perspective?

관점은 질문의 검색 공간을 바꾼다

같은 “2022 동계올림픽 개막식”이라도 일반 독자는 장소와 날짜를 묻지만, 이벤트 기획자는 교통, 예산, 참가국 입장 순서 같은 운영 질문을 떠올린다. STORM은 이런 역할 차이를 의도적으로 만든다.

결과적으로 검색 쿼리도 넓어지고, 최종 개요가 표면 사실을 넘어 세부 섹션을 더 많이 포함한다.

03
Evaluation Design

평가를 어떻게 설계했나

논문은 최종 글만 보지 않고, 사전 조사 결과인 개요 품질도 따로 평가한다. STORM의 목표가 “글쓰기 전 단계” 자동화이기 때문이다.

Dataset
FreshWiki

최근 Wikipedia 글 중 품질 기준을 통과한 문서를 모아 평가셋을 구성했다.

Outline Metrics
2개

heading soft recall과 heading entity recall로 사람이 만든 목차와 얼마나 겹치는지 본다.

Article Metrics
7개

ROUGE, 엔티티 회수율, 흥미도, 조직성, 관련성, 커버리지, 검증가능성을 평가한다.

Human Eval
10명

숙련 Wikipedia 편집자가 STORM과 oRAG 결과를 비교 평가했다.

Baselines

비교 대상

Direct Gen
외부 검색 없이 바로 개요와 글 생성
RAG
주제 검색 결과를 넣고 개요 또는 글 생성
oRAG
개요를 만든 뒤 섹션 제목으로 추가 검색해 섹션별 작성
STORM
관점별 질문 대화로 자료를 모은 뒤 개요와 글 생성
What Is Fair?

평가의 핵심 비교

oRAG도 개요 기반 RAG라 강한 기준선이다. 따라서 STORM이 oRAG를 이긴다는 것은 단순히 “개요가 있어서 좋다”가 아니라, 개요를 만들기 전 질문과 조사 방식이 추가 가치를 만든다는 뜻이다.

논문은 또한 perspective 제거, conversation 제거, outline stage 제거 실험으로 어떤 부품이 성능에 기여하는지 분리했다.

04
Results

결과는 무엇을 말하나

자동 평가는 STORM이 더 넓은 개요와 더 풍부한 글을 만든다고 보고한다. 사람 평가는 조직성과 커버리지 향상을 확인하면서도, 검증가능성은 아직 충분히 해결되지 않았다고 지적한다.

05
Limitations

중요한 한계와 해석

STORM은 좋은 초안을 만드는 데 유용하지만, 곧바로 게시 가능한 Wikipedia 글을 만든다는 뜻은 아니다. 논문 자체도 편집자 피드백을 통해 여러 위험을 강조한다.

Source Bias

출처 편향이 옮겨온다

검색 엔진과 웹 문서가 특정 관점에 치우치면, 생성된 글도 그 어조와 편향을 따라갈 수 있다.

Over-association

근거 사이를 과하게 연결한다

존재하는 사실을 가져오더라도 서로 무관한 사실을 그럴듯하게 연결하면 검증가능성이 떨어진다.

Not Publication-ready

사람 글보다 아직 부족하다

편집자들은 STORM 결과가 유용한 시작점이라고 보았지만, 잘 다듬어진 인간 작성 문서에는 못 미친다고 평가했다.

실무 적용 시 결론 STORM은 “최종 답변 생성기”보다 “리서치 어시스턴트”로 쓰는 편이 안전하다. 사람이 개요, 출처 균형, 논리적 연결, 중립성을 검토하는 단계를 반드시 남겨야 한다.
06
LangGraph Angle

LangGraph로 보면 어떻게 나뉘나

STORM은 상태를 축적하며 여러 관점의 대화를 병렬로 돌리는 구조라 LangGraph 예제로 옮기기 좋다. 핵심 상태는 topic, perspectives, conversations, references, draft_outline, final_outline, article_sections다.

node 01
survey_related_topics

유사 주제와 목차를 모아 관점 후보의 재료를 만든다.

node 02
generate_perspectives

기본 사실 작성자와 전문 관점 작성자를 만든다.

parallel
run_interviews

관점별 writer-expert 대화를 병렬 실행한다.

loop
retrieve_and_answer

질문 분해, 검색, 출처 필터, 답변 합성을 반복한다.

node 05
refine_outline

초안 개요와 대화 기록을 합쳐 최종 개요를 만든다.

node 06
write_sections

섹션별 관련 참고자료를 골라 인용 있는 글을 생성한다.

Graph Pattern

fan-out / fan-in이 핵심

관점별 인터뷰는 독립적이므로 fan-out으로 병렬 처리하고, final outline 생성에서 fan-in으로 합친다. 이 구조는 LangGraph의 조건부 edge와 map-reduce 패턴에 잘 맞는다.

Guardrails

검증 노드를 추가해야 한다

논문의 한계를 줄이려면 source diversity check, claim-source entailment check, unsupported inference detector 같은 검증 노드를 별도로 두는 편이 좋다.
